## 1. Терминологическая развилка

**«Маломобильные граждане»** в российском нормативном понимании — это ОВЗ опорно-двигательного аппарата (передвижение в кресле-коляске, с тростью, костылями). Для веб-интерфейса ЭИОС напрямую малозначимо: с экраном они работают как остальные. Их специфика — на уровне физической доступности здания, аудиторий, расписания.

**Применимая для ЭИОС категория** — **обучающиеся с инвалидностью и ОВЗ** в смысле статьи 79 ФЗ «Об образовании в РФ». Реально включает: слабовидящих, незрячих, слабослышащих, глухих, людей с нарушением моторики, с когнитивными особенностями, с эпилепсией.

В этом документе под «доступностью» понимается **доступность для всех перечисленных категорий**, не только мобильно ограниченных.

## 2. Нормативная база

### 2.1. Базовый стандарт

**ГОСТ Р 52872-2019** — российская адаптация **WCAG 2.1**. Уровень соответствия — **AA**.

Для Дидактикона дополнительно берутся отдельные критерии **WCAG 2.2** (не предписаны ГОСТом, но критичны для PWA-паттернов):
- **2.4.11 Focus Not Obscured** (AA) — фокус не должен уезжать под sticky-элементы.
- **3.3.8 Accessible Authentication** (AA) — не блокировать paste в пароль, поддержать passkeys.

Также берётся критерий AAA **2.5.5 Target Size 44×44** как более строгая версия требования к размеру тап-цели (см. `accessibility-architecture-policy.md`, §7.3 и §7.6). Критерий 2.5.8 (AA, 24×24) выполняется как следствие.

### 2.2. Российская нормативная база

- **ФЗ-273, ст. 79** — обязывает обеспечить условия обучения для лиц с ОВЗ.
- **Приказы Минобрнауки об аккредитации** — учитывают доступность ЭИОС как **проверяемый показатель**.

### 2.3. Декларация соответствия

> Дидактикон публикует декларацию доступности по стандартизованному URL **`/accessibility`** относительно домена развёртывания.

Содержание страницы:
- уровень соответствия (AA);
- перечень разделов ГОСТ Р 52872-2019, по которым достигнуто соответствие;
- известные ограничения;
- **раздел «Почему отсутствует отдельная версия для слабовидящих»** (см. ниже §2.4 и `accessibility-architecture-policy.md`, §1.2.1);
- канал обратной связи о проблемах доступности (форма);
- дата последней оценки;
- дата следующей плановой переоценки.

Это **формальная точка проверки** аккредитационной комиссией и реальная точка обратной связи для пользователей. Подробнее — `accessibility-architecture-policy.md`, §7.5.

### 2.4. Архитектурное ограничение: одна версия

> **Дидактикон принципиально не реализует «версию для слабовидящих» как отдельный интерфейс.** Доступность обеспечивается единой кодовой базой, соответствующей ГОСТ Р 52872-2019.

Высокая контрастность реализуется **как тема внутри общего интерфейса**, не как отдельный сайт.

#### 2.4.1. Защита решения в декларации

Это решение должно быть **явно обосновано** в публичной декларации `/accessibility`. Действующий ГОСТ Р 52872-2019 не требует отдельной версии — он требует доступности самого интерфейса на уровне AA. Однако предыдущая редакция (ГОСТ Р 52872-2012) интерпретировалась судами и проверяющими как требование размещать «версию для слабовидящих», и эта инерция надзорной практики реальна.

Поэтому раздел декларации содержит:
- ссылку на действующую редакцию ГОСТ 2019 и её отличие от редакции 2012;
- архитектурное обоснование единого доступного интерфейса;
- разъяснение, что соответствие AA обеспечивает доступность для слабовидящих в рамках единой кодовой базы.

Полное обоснование политики и текст для декларации — в `accessibility-architecture-policy.md`, §1.2.1.

## 3. Принцип: закладываем сразу

Поздняя адаптация = переписывание половины компонентов и палитры. Ранняя = пара процентов накладных расходов на разработку.

**В MVP Дидактикона базовые требования AA выполняются по умолчанию.** Доступный интерфейс — это просто хороший интерфейс для всех. Семантический HTML, клавиатурная навигация, корректный контраст, разумные размеры тап-целей, текстовые альтернативы — это базовая инженерная гигиена.

Специальные режимы (высокий контраст, увеличенный шрифт) — это **надстройка** над доступной базой.

## 4. Категории пользователей и их потребности

### 4.1. Слабовидящие
Самая массовая категория в вузе. Масштабирование шрифта без поломки вёрстки, контраст, отсутствие мелких целей, читаемая типографика.

### 4.2. Незрячие
Меньшинство по численности, но критичная категория — если не сделать, **полностью отрезаны от системы**. Полная навигация скринридером (NVDA, JAWS, VoiceOver, TalkBack), семантический HTML и ARIA, текстовые альтернативы.

### 4.3. Слабослышащие и глухие
Субтитры и расшифровки для аудио/видео в материалах пакетов, отсутствие звуковых сигналов без визуальных дублей.

### 4.4. С нарушением моторики
Полная клавиатурная навигация, крупные тап-цели, отсутствие требований к точности и скорости (жесты, drag-and-drop), отмена случайных действий.

### 4.5. С когнитивными особенностями (дислексия, СДВГ, РАС)
Простой язык, отсутствие мигания и автоматических анимаций, предсказуемая навигация, ясная структура, отсутствие давления времени без альтернатив.

### 4.6. С эпилепсией
Запрет мигающего контента >3 вспышек в секунду — медицинский риск.

## 5. Технические требования

> **Замечание о числовых порогах.** Все конкретные значения (контраст, размеры тап-целей, размеры шрифтов, межстрочные интервалы, пороги мигания) синхронизированы с **нормативным реестром порогов** в `accessibility-architecture-policy.md`, §7.6. Если значение в этом документе расходится с реестром — реестр является источником истины. При плановом обновлении политики числовые пороги меняются **только в реестре**, текст этого документа правится следом.

### 5.1. Семантика и структура

- **Семантический HTML.** `<nav>`, `<main>`, `<article>`, `<section>`, `<aside>`, `<button>`, `<a>`. **Не** `<div onclick>`.
- **ARIA-разметка** для кастомных компонентов через дизайн-систему (Radix UI / Headless UI как референс).
- **Landmark-роли** на уровень страницы — скринридер должен уметь перепрыгнуть через шапку сразу к контенту.
- **Заголовочная иерархия** строгая: `<h1>` один на странице, без пропусков уровней.
- **Skip-link** «Перейти к основному содержимому» — невидимая ссылка в начале tab-order, появляющаяся при фокусе.

### 5.2. Атрибут `lang`

- **`<html lang="ru">`** обязательно. Без этого скринридер читает русский английским голосом.
- **При переключении локали** в настройках (`ru` / `en` / `kk-Cyrl`) — `lang` на `<html>` меняется реактивно.
- **Иноязычные фрагменты внутри страницы** — оборачиваются в элемент с собственным `lang` (например, английская цитата в русской лекции — `<span lang="en">...</span>`).

### 5.3. SPA-навигация и управление фокусом

> **Самое критичное требование для PWA-архитектуры.**

При client-side навигации (роутер React/Vue/Svelte) скринридер **не знает**, что страница сменилась. Без явных мер незрячий пользователь теряет ориентацию.

На каждый переход маршрута:
- **Обновлять `document.title`** на осмысленный заголовок страницы (WCAG 2.4.2, A).
- **Переносить фокус** на `<h1>` нового экрана или на `<main>`, ИЛИ анонсировать смену через `aria-live="polite"`.
- **При удалении элемента из списка / подгрузке нового контента** — фокус не должен теряться в `<body>`. Управление фокусом — явное.
- **`aria-busy="true"`** на контейнере во время загрузки, не только спиннер.

Это **не решается компонентной дизайн-системой** — это уровень роутера и интеграции с ним.

### 5.4. Клавиатурная навигация

- **Все интерактивные элементы достижимы Tab.**
- **Видимый focus-ring** на всём фокусируемом. Tailwind `focus-visible:` варианты. **Не убирать outline стилем.**
- **Tab-order логичен** — слева-направо, сверху-вниз, по визуальному порядку.
- **Esc закрывает** модалки, шторки, выпадашки.
- **Enter** активирует кнопки и ссылки. **Space** — кнопки и чекбоксы.
- **Стрелки навигируют внутри компонентов** (табы, меню, радиогруппы) — но не между несвязанными элементами.
- **Focus trap в модалках** — Tab не уводит за пределы. При закрытии фокус возвращается на элемент, открывший её.

### 5.5. Focus Not Obscured (WCAG 2.4.11, 2.2 AA)

> **Прямой удар по mobile-first паттернам Дидактикона.**

Sticky-шапка, sticky bottom-tabs, bottom sheets — это ровно ситуации, где сфокусированный элемент уезжает под липкий элемент или за край шторки и становится невидим.

Требования:
- **`scroll-margin-top`** на фокусируемых элементах, равный высоте sticky-шапки (включая compact-mode).
- **`scroll-margin-bottom`** аналогично для bottom-tabs.
- **scroll-into-view с offset** при программной прокрутке к элементу.
- **При открытии bottom sheet** — проверять, что фокус не за её краем.

### 5.6. Контраст и цвет

- **Контраст основного текста ≥ 4.5:1**, крупного (≥ 18pt или ≥ 14pt жирный) ≥ 3:1. Уровень AA.
- **Контраст границ интерактивных элементов ≥ 3:1.**
- **Не полагаться на цвет как единственный признак.** Красный долг — это не только красный, ещё иконка восклицания. Зелёный успех — не только зелёный, ещё галочка.
- **Артефакт обязателен:** **протестированная матрица контраста токенов** дизайн-системы для светлой и тёмной тем. Не «на глаз», а измеренные пары. Граничные токены (типа `gray-500` на `gray-900`) могут провалить 4.5:1 — проверять реально.
- **Halation в тёмной теме.** Чисто белый (`#fff`) на чисто чёрном (`#000`) даёт ореол у пользователей с астигматизмом. Использовать **`gray-50` / `gray-100`** для основного текста на тёмно-сером фоне (`gray-900` / `gray-950`), а не `#fff` на `#000`.
- **Высокая контрастность** как отдельная тема — после MVP, но архитектурно не закрывать дверь (всё через CSS-переменные).

### 5.7. Шрифт и типографика

- **Размер шрифта в `rem`**, не `px`.
- **Базовый размер ≥ 16px** (`1rem = 16px`).
- **Межстрочный интервал ≥ 1.5** для длинного текста.
- **Никакого `text-align: justify`** — растягивает пробелы, плохо дислексикам.
- **Шрифт без декоративных засечек** для UI. Inter, Roboto, **Atkinson Hyperlegible**.
- **Atkinson Hyperlegible** — рассмотреть как опцию шрифта в настройках после MVP. **Проверить полноту кириллицы** перед включением — шрифт может давать странную высоту строки на русских и казахских текстах.
- **CSS для дислексии** как опция (после MVP): увеличенный межбуквенный (`letter-spacing: 0.12em`) и межсловный (`word-spacing: 0.16em`) интервалы. Включается отдельной настройкой профиля или вместе с режимом «крупный шрифт».
- **Text Spacing (WCAG 1.4.12, AA).** Пользователь должен мочь увеличить line-height / letter-spacing / word-spacing своим стилем без обрезки текста. **Следствие для компонентов:** никаких фиксированных `height` на текстовых контейнерах — только `min-height`.

### 5.8. Цели тапа и взаимодействие

- **Минимальный размер тап-цели — 44×44 CSS-пикселя** (WCAG 2.5.5 AAA). Берём выше AA-минимума (24×24, WCAG 2.5.8) — критично для mobile-first и людей с тремором рук.
- **Расстояние между интерактивными элементами ≥ 8px.**
- **Свайпы дублируются кнопками** (уже в концепции UI). Нельзя сделать действие **только** жестом.
- **Drag-and-drop дублируется** кнопками «вверх / вниз», «перенести в …».
- **Никаких таймаутов** без возможности продлить или отменить.

### 5.9. Жесты, зум, ориентация

- **Запрет на `user-scalable=no`** и `maximum-scale=1` в viewport-meta. WCAG 1.4.4 / 1.4.10 — FAIL. Часто прописывается в дефолтных шаблонах мобильных PWA «чтобы не зумилось при двойном тапе» — **это нарушение**.
- **Не лочить ориентацию.** WCAG 1.3.4 (AA). Некоторые пользователи держат планшет в фиксированном креплении.
- **Минимум 320 CSS-пикселей** по ширине без горизонтального скролла (WCAG 1.4.10).

### 5.10. Альтернативный контент

- **`alt` для всех `<img>`.** Декоративные — `alt=""` явно.
- **`aria-label` для всех иконочных кнопок** (закрытие, бургер, свайп-экшены). Без подписи на экране каждой нужна программная подпись.
- **Label in Name** (WCAG 2.5.3). Если у кнопки видимая подпись «Войти» — её доступное имя должно начинаться с «Войти» (голосовое управление ищет по видимому тексту).
- **Подписи для всех полей формы.** **Не плейсхолдер вместо лейбла.**
- **`autocomplete`** на полях имени, email, телефона, паролей (WCAG 1.3.5, AA). Помогает моторике и когнитивке через автозаполнение.
- **Сообщения об ошибках формы — текстом, не только цветом**, с конкретикой.
- **`aria-describedby`** связывает поле с описанием ошибки.

### 5.11. Authentication (WCAG 3.3.8, 2.2 AA)

- **Paste в поле пароля должен работать.** Запрет paste ломает менеджеры паролей — главный механизм для людей с нарушением моторики и памяти.
- **Поддержка passkeys / WebAuthn** — там, где OIDC-провайдер (Универкон) это поддерживает.
- **Magic-link** как альтернатива паролю — где политика ОО разрешает.
- **CAPTCHA — без визуальных головоломок.** Только невидимая (honeypot, reCAPTCHA v3) или с аудио-альтернативой. Аудио-CAPTCHA должна быть реально разборчива.

### 5.12. Динамические уведомления

- **ARIA live regions.** `aria-live="polite"` для обычных, `aria-live="assertive"` для критичных.
- **Не только тост с автоисчезновением.** Альтернатива: постоянное место для сообщений или возможность пересмотреть последние.
- **Звуковые сигналы дублируются визуальными.**

### 5.13. Движение и анимация

- **`prefers-reduced-motion`** отключает все необязательные анимации.
- **Никакого мигания** >3 вспышек в секунду.
- **Автопроигрывание видео/аудио — запрещено**.
- **Бесконечные анимации** (вращающиеся спиннеры) — короткие или с возможностью паузы.

### 5.14. Табличные данные

> **Большая дыра для LMS.** Универкон — это академический учёт: ведомости, расписание, посещаемость, оценки. Это таблицы по природе.

- **`<table>` с `<th scope="col">` / `<th scope="row">`**, не «layout-таблицы через CSS Grid из `<div>`».
- **`<caption>`** с описанием таблицы.
- **Заголовки строк и столбцов корректно связаны** с ячейками.
- **Сложные таблицы** (с rowspan/colspan, многоуровневыми заголовками) — `headers` / `id` для явных связей.
- **Стрелки навигации внутри таблицы** в режиме `role="grid"`.
- **Текстовые альтернативы графиков и диаграмм.** Если показатель отображается Canvas / SVG-диаграммой — обязательно дублирование таблицей или текстовой сводкой.

### 5.15. Календарь и расписание

Мини-календарики часто делаются `<div>`-ами с CSS Grid. **Обязателен table-паттерн** или `role="grid"` с навигацией стрелками.

Карточки событий в расписании — фокусируемы, читаются скринридером в осмысленном порядке (дата → время → дисциплина → ведущий → помещение, а не как разложены визуально).

### 5.16. Тесты и оценивание (специфика образования)

- **Drag-and-drop в вопросах** обязан иметь клавиатурную альтернативу — выпадающие списки, чекбоксы, кнопки «вверх/вниз».
- **Выбор области на изображении** (hot-spot) — альтернатива в виде списка областей с подписями.
- **Таймеры тестов.** Если таймер ограничивает прохождение, пользователь с `accessibility_preferences` должен иметь возможность **запросить продление** до начала теста, без раскрытия причины.
  - Технически: флаг «нужно больше времени» в профиле → система применяет коэффициент к лимиту теста.
  - Коэффициент — настройка ОО (типично 1.5× или 2×).
  - Запрос анонимен для педагога: видно только «время продлено по запросу студента», без указания причины.

### 5.17. Понятный язык (WCAG 3.1.5, AAA — берётся в политику)

Системные сообщения — на понятном языке.

**Антипример:** «Аутентификационный токен протух».
**Корректно:** «Сеанс истёк, войдите снова».

Для русского языка: избегать сложных деепричастных оборотов и канцелярита в интерфейсных подсказках. Это требование к **разработчикам**, текстам в UI. Контент пакетов уроков — забота педагога, не платформы.

### 5.18. QR-коды

QR на дверях помещений (см. `didakticon-event-sharing-room-qr.md`) бесполезен для незрячих — они физически не могут навестить камеру.

- **Под каждым QR-стикером** — читаемая ссылка текстом (короткий URL, см. концепцию QR помещений).
- **Тактильный код помещения** на стикере (для незрячих — пальцем найти и попросить помощника прочитать) — рекомендация ОО, не Дидактикона, но влияет на дизайн стикера.

### 5.19. Мультимедиа в пакетах уроков

Требования к **манифесту пакета**, не к плееру (детально — в edu-framework-domain-merge-map.md, п. 7.4):

- **Аудио — обязательная текстовая расшифровка.**
- **Видео — субтитры обязательны, аудиодескрипция желательна.** Whisper для русского с вычиткой педагогом.
- **Сурдоперевод** — не обязателен, плюс при работе с глухими студентами.

### 5.20. Сторонние интеграции (iframe)

Если Дидактикон встраивает сторонние сервисы (YouTube-плеер, доска Miro, видеоконференция, SCORM-пакет):

- **Ответственность за доступность стороннего компонента — на стороне ОО**, выбирающего конкретный сервис. Дидактикон **обязан** обеспечить корректное обрамление: `<iframe title="...">`, фокусируемость, возможность выхода из iframe Tab-ом.
- **SCORM-пакеты** имеют непредсказуемый уровень доступности. В декларации пакета `wcag_level` обычно `unknown`. Это не блокер импорта, но фиксируется явно.

### 5.21. Сетевые состояния и offline

PWA-специфика. Когда сеть отвалилась или ответ медленный:
- Не молчать. Объявлять состояние через `aria-live`.
- Не показывать спиннер бесконечно — после N секунд переходить в состояние «медленная сеть» с явным текстом.
- В offline-режиме — явное сообщение «нет связи, показаны данные на момент Х», читаемое скринридером.

## 6. Тестирование

### 6.1. Статический анализ (shift-left)

Проблемы дешевле всего ловить **до коммита**.

- **`eslint-plugin-jsx-a11y`** для React в конфиге проекта. Уровень — `error`, не `warn`. Без него в ревью проскакивают `<div onClick>` и иконочные кнопки без `aria-label`.
- **pre-commit hook** с линтером — компилируется и проверяется только изменённый код, быстро.
- **`stylelint-a11y-plugin`** — для CSS-правил (контраст, размеры).

### 6.2. Автоматическое тестирование (CI)

- **axe-core** в pipeline на каждом PR. Блокирует слияние при нарушениях `serious` / `critical`.
- **Lighthouse Accessibility audit** на ключевых страницах — минимум на каждый релиз.
- **Storybook a11y addon** — каждый вариант каждого компонента проверяется панелью Accessibility.
- Это **не заменяет** ручное тестирование, но отлавливает регрессии и недопустимый базовый уровень.

### 6.3. Ручное тестирование

- **Прогон по клавиатуре** всех ключевых флоу: вход → дашборд → открытие занятия → тест → выход. Без мыши вообще.
- **Прогон со скринридером.** Минимум: NVDA (Windows), VoiceOver (iOS). Желательно: JAWS (Windows), TalkBack (Android).
- **Симуляция дальтонизма** (Chrome DevTools → Rendering → Emulate vision deficiencies).
- **Тестирование при увеличении 200%** в браузере.
- **Тестирование с увеличенным шрифтом ОС** (Android Display size).

### 6.4. Пользовательское тестирование

Привлечь реальных пользователей с ОВЗ из числа студентов ОО. Это работает лучше любого автомата.

**Протокол модерируемого тестирования:**
- Не просить «потестируйте», а дать **конкретный сценарий**:
  - «Войдите в систему, найдите расписание на завтра, откройте материалы первой пары, скачайте файл задания».
  - «Откройте тест по дисциплине X, начните попытку, ответьте на первые три вопроса, сохраните и закройте».
- Тестировать **с закрытыми глазами на десктопе** (VoiceOver / NVDA) — быстро вскрывает проблемы пропущенных лейблов и порядка чтения.
- Координация — через центр инклюзивного образования ОО (если есть).
- Оплата участникам — нормальная практика (стипендия или договор ГПХ). Бюджет на это закладывается **ОО**, не командой разработки.

### 6.5. Мониторинг продакшена

Real User Monitoring (RUM) — собирать обезличенные события:
- процент сессий с включённым скринридером (определяется по специфичным DOM API);
- процент с `prefers-reduced-motion`;
- процент пользователей увеличенного шрифта;
- процент клавиатурных сессий (без mousemove).

**Назначение:**
- аргумент бизнесу: «17% студентов используют увеличенный шрифт — функцию нельзя выпиливать»;
- раннее обнаружение деградации (внезапный рост ошибок навигации у скринридер-пользователей).

**Без слежки:** агрегированно, без поимённой привязки.

## 7. Расширения через домены edu-framework

Доступность не решается только в Дидактиконе. Часть требований уходит во фреймворк — добавления к карте `edu-framework-domain-merge-map.md`.

### 7.1. Идентичность и контекст

**Новое поле физлица (опциональное):**

```
Физлицо:
  accessibility_preferences: {
    font_size: enum {small, normal, large} | null,
    high_contrast: bool | null,
    reduced_motion: bool | null,
    screen_reader: bool | null,  // саморепортируемый
    dyslexia_friendly: bool | null,
    extended_test_time: bool | null  // флаг «нужно больше времени на тестах»
  } | null
```

**Назначение:** один раз настроил — получил на любом устройстве и в любом приложении ОО.

**Юридически:** это **не медицинские данные**, см. `accessibility-architecture-policy.md`, п. 8.2. Только саморепортируемые предпочтения интерфейса.

### 7.2. Аудиторный фонд

**Атрибут помещения:**

```
Помещение:
  accessibility: {
    wheelchair_access: bool,
    elevator_nearby: bool,
    induction_loop: bool,        // индукционная петля
    tactile_signage: bool,       // тактильная навигация
    accessible_restroom_nearby: bool
  }
```

Расписание может учитывать ограничения студента; студент видит атрибуты в карточке события.

### 7.3. Академический контур

Адаптированные образовательные программы (АОПОП, АОП) — задача академического контура. **Не делать атрибут «эта группа — для ОВЗ» на уровне события** — это сегрегация. ОВЗ — атрибут студента, не события; адаптация — индивидуальная.

**Продление времени теста** через флаг `extended_test_time` в `accessibility_preferences` — реализуется на уровне Тестикона. Коэффициент продления — настройка ОО.

### 7.4. Пакеты уроков

**Декларация доступности в манифесте** (см. карту вливания):
- `has_captions`, `has_transcript`, `has_audio_description`, `sign_language`;
- `wcag_level: {fail, A, AA, AAA, unknown, not_applicable}`;
- Заполнение **обязательно** при публикации пакета. Для собственного контента ОО:
  - `not_applicable` — допустимо для отдельных пунктов с обоснованием (не вся медицинская атласная картинка имеет смысловую альтернативу);
  - **`unknown` запрещено** — это маркер «не сделали»;
- Для **импортируемого стороннего контента** (SCORM-пакеты внешних вендоров, OER-материалы Open edX, видеоэмбеды): **`unknown` допустимо как честная декларация** — платформа не может гарантировать доступность вложенного содержимого. См. `accessibility-architecture-policy.md`, §2.4.1.

**Accessibility Score пакета** (после MVP) — автоматический агрегированный показатель с разбивкой по пунктам. Перерасчёт при каждом изменении контента. Полное определение метрики с защитой от Goodhart-эффекта — в следующей итерации политики.

### 7.5. Адресные коммуникации (D??)

- **Дублирование важных уведомлений** в визуальном канале для глухих, в аудио — для незрячих (через системные средства).
- Это не новые жанры, а **правила формирования сообщений** в зависимости от `accessibility_preferences` адресата.

### 7.6. Документы (новая проекция фреймворка)

**Tagged PDF, корректный reading order, текстовые альтернативы** — требование к рендер-движкам всех продуктов экосистемы, не только Дидактикона.

Это часто **главный провал инклюзии**: ЭИОС в UI доступна, а скачанная ведомость — картинка без структуры.

Регулируется на уровне рендера документов. Для Дидактикона: все экспорты (расписание в PDF, материалы пакета в PDF) — tagged.

### 7.7. Политика исключений (недоступный контент)

> **Реестр исключений.** У любого материала может быть статус доступности **«Не применимо / Недоступно / Частично доступно»** с обоснованием.

Если материал недоступен:
- Дидактикон **предлагает запросить альтернативный формат** кнопкой на странице материала.
- Запрос направляется автору контента (педагогу) и в копию — в центр инклюзивного образования ОО (если есть).
- Студент **не обязан указывать причину**. Достаточно описания нужного формата.
- ОО ведёт реестр обработанных запросов — это защищает при аккредитационной проверке (п. 5.1.3.3 ГОСТ Р 52872-2019).

### 7.8. Канал обратной связи

> **На странице `/accessibility` — форма «Сообщить о проблеме доступности».**

- Поля: что не работает, на каком экране, какое вспомогательное средство используется.
- Опционально: контакт для связи.
- Получатель: команда продукта + центр инклюзии ОО.
- **SLA на ответ** — фиксируется ОО, типично 5 рабочих дней.

Этот канал — реальный механизм исправления, а не формальность.

### 7.9. AI-помощь автору контента

Расширение Метакона и Дидактикона: при загрузке материала педагогом — автоматический accessibility-чек:
- **Изображение без `alt`** → AI предлагает описание, педагог редактирует.
- **Видео без субтитров** → автоматическая генерация (Whisper) с пометкой «требует вычитки».
- **Аудио без расшифровки** → автоматическая транскрипция.
- **Таблица без заголовков** → предложение разметить.
- **Сложный текст** → предложение упростить (опционально, без давления).

Это **очень в духе философии Дидактикона** — платформа помогает педагогу делать качественный продукт, не наказывает.

### 7.10. Сквозной паттерн фреймворка

Дополнение к карте вливания (`edu-framework-domain-merge-map.md`):

**Паттерн «accessibility declaration»** — любая сущность фреймворка, представляющая контент для пользователя (учебное событие, материал пакета, объявление Хроникона, документ Кабитория), имеет стандартизированную декларацию доступности с единой формой записи.

## 8. Прагматика: что делаем когда

### 8.1. В MVP (накладные расходы — единицы процентов разработки)

**Базовая платформа:**
- Семантический HTML.
- `lang` атрибут с реактивной сменой.
- Клавиатурная навигация и focus-visible.
- **SPA-навигация с управлением фокусом** при смене маршрута.
- **Запрет на `user-scalable=no`** в viewport.
- **Focus not obscured** при sticky-шапке и bottom-tabs (`scroll-margin`).
- ARIA для кастомных компонентов через **Radix UI / Headless UI**.
- Skip-link, landmark-роли.
- Контраст по AA в обеих темах, с тестированной матрицей токенов.
- **Halation-safe** тёмная тема (не #fff на #000).
- `rem`-шрифты и размер в настройках.
- `prefers-reduced-motion`.
- Минимум **44×44** для тап-целей.
- **Text Spacing AA**: `min-height` вместо `height` в компонентах.
- Дублирование цвета иконкой/паттерном.
- `alt`-тексты, лейблы форм, `aria-label` для иконочных кнопок.
- `autocomplete` на полях профиля.
- **Семантика таблиц** для ведомостей, расписания, оценок.
- **Accessible authentication**: paste в пароль работает, никаких визуальных CAPTCHA.

**Процессы:**
- **eslint-plugin-jsx-a11y** в конфиге (level: error) и pre-commit hook.
- **axe-core** в CI с блокировкой слияния PR при `serious` / `critical`.
- **Storybook a11y addon** на компонентной библиотеке.
- **Decoration of Done с a11y-чеклистом** (см. политику, п. 5).

**Артефакты:**
- Декларация по URL **`/accessibility`**.
- Форма «Сообщить о проблеме доступности».
- **Архитектурное ограничение «одна версия»** зафиксировано в публичной документации продукта.

### 8.2. После MVP

- **Онбординг настроек доступности** при первом входе нового пользователя (карточка «Хотите настроить интерфейс?» с быстрыми пресетами).
- Режим высокого контраста как отдельная тема.
- Atkinson Hyperlegible как опция шрифта (после проверки кириллицы).
- **CSS-настройки для дислексии** (letter-spacing, word-spacing).
- Полное тестирование со скринридерами на всех экранах.
- Синхронизация `accessibility_preferences` с Универконом.
- Декларация доступности в манифесте пакетов (после расширения домена).
- Учёт `accessibility` помещений в расписании.
- **AI-помощь автору контента** (alt, субтитры, транскрипции, чек таблиц).
- **Accessibility Score пакета** — агрегированный показатель.
- **RUM-мониторинг** использования a11y-фич.
- **Версионирование декларации** доступности — автоматический пересчёт при изменении контента пакета.

### 8.3. Зона ответственности ОО

- Сурдоперевод видеоматериалов.
- Адаптированные учебные программы (АОПОП).
- Физическая доступность помещений (пандусы, лифты, навигация).
- Tagged PDF в шаблонах документов от авторов.
- Обучение педагогов созданию доступного контента.
- Бюджет на пользовательское тестирование с участниками с ОВЗ.
- Центр инклюзивного образования и его процессы.
- SLA на ответы по форме обратной связи.

### 8.4. Не делаем

- Голосовое управление приложением.
- Brailleterminal-вывод (поддерживается скринридерами через системные API).
- AAA-соответствие на всех экранах (только AA + выборочные AAA).
- Отдельная «версия для слабовидящих» — архитектурно запрещено.
- Хранение медицинских данных в Дидактиконе.

## 9. Открытые вопросы / TODO

1. **Хранение `accessibility_preferences`** — local в MVP, миграция в Универкон после расширения домена идентичности.
2. **Деликатные данные.** Дидактикон не хранит и не оперирует данными о состоянии здоровья (см. политику, п. 8). Если студент захочет загрузить справку для предоставления льгот — это **в Кабиторий**, не в Дидактикон. Это снижает требования к защите системы и стоимость её аттестации.
3. **Атрибут доступности помещения** — кто заполняет в Универконе? АХЧ при паспортизации зданий? Требует процессного решения на стороне ОО.
4. **Доступность пакетов уроков** — обязательная декларация с возможностью «not_applicable» для каждого пункта, но не «unknown» (кроме SCORM-импорта).
5. **Аудит** существующей кодовой базы перед релизом. axe-core + jsx-a11y с первого спринта — без этого технический долг копится незаметно.
6. **Привлечение реальных пользователей с ОВЗ** для бета-тестирования. Координация с центром инклюзии ОО.
7. **Методичка для педагогов** «как сделать доступный пакет урока» — короткий документ. Уровень содержания, не интерфейса.
8. **Декларация соответствия ГОСТ Р 52872-2019** — формальный документ, прилагаемый к ЭИОС при аккредитации.
9. **Коэффициент продления времени тестов** — настройка ОО? Дисциплины? Единый для всех с флагом `extended_test_time`? Требует решения архитектора Тестикона.
10. **Что делать с тестами, где визуальное распознавание — часть проверяемого умения?** (Например, тест по гистологии — рассмотреть микрофотографию). Сегодня это блокер для незрячих студентов. Решение — на стороне ОО (адаптированная образовательная программа), не на стороне платформы.
11. **Tagged PDF в Универконе.** Расширение рендера документов. Отдельная задача 1С-разработки, не Дидактикона. Должна быть синхронизирована по срокам.
12. **Совместимость Radix UI / Headless UI с PWA-режимом** — проверить на iOS Safari, особенно с focus management в модалках.
