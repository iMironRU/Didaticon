# Доступность. Архитектурная политика edu-framework

**Статус:** v2, закрытие багов первого черновика. Структурный сдвиг (cross-cutting concern, Non-goals, аксиомы, метрики, миграция, обучение) — отдельной итерацией.
**Область:** вся экосистема Univerkon / edu-framework (Универкон, Дидактикон, Кабиторий, Хроникон, Лицензикон, Ярлыкон, Тестикон, Росткон и любые будущие продукты под брендом).
**Назначение:** высокоуровневая политика доступности. Описывает **принципы и инварианты**, не конкретные технические требования. Технические требования — в специфических документах продуктов (например, `didakticon-accessibility.md`).
**Связанные документы:** `edu-framework-domain-merge-map.md`, `didakticon-accessibility.md`, и все доменные спецификации фреймворка.

## 1. Концептуальная рамка

### 1.1. Доступность — инвариантное свойство системы

> **Accessibility by Design.** Доступность не является функцией, режимом, опцией или специальным интерфейсом. Это **инвариантное свойство** любого продукта экосистемы Univerkon, такое же как корректность, безопасность, надёжность.

Следствия:
- Любой новый компонент, экран, пакет уроков, сервис, микрофронтенд, документ или API-контракт считается **неполным**, пока не удовлетворяет минимальным требованиям доступности.
- Доступность не имеет статуса «сделаем потом». Компонент без доступности — не «MVP», а **технический долг**.
- Доступность не имеет статуса «частная задача отдельного человека». За доступность отвечает каждый автор спецификации, кода или контента.

### 1.2. Один интерфейс для всех

> **Отказ от «специальных версий».** Экосистема Univerkon принципиально **не реализует** «версии для слабовидящих», «упрощённые интерфейсы», «отдельные мобильные сайты для инвалидов» и аналогичные сегрегированные продукты.

Обоснование:
- Сегрегированные интерфейсы хуже основного по функциональности — их меньше тестируют, медленнее обновляют, ими меньше пользуются.
- Сегрегация конфликтует с инклюзивной образовательной политикой ФЗ-273.
- Один корректно спроектированный интерфейс **технически проще, функционально полнее и юридически корректнее**, чем «обычный + альтернативный».

Все пользователи работают с одной и той же системой. Различия — на уровне **пользовательских предпочтений** (размер шрифта, контраст, скринридер), не на уровне продукта.

#### 1.2.1. Защита решения от правоприменительной инерции

Предыдущая редакция **ГОСТ Р 52872-2012** на практике трактовалась судами и надзорными органами как требование размещать на сайтах организаций «версию для слабовидящих» — ровно тот сегрегированный антипаттерн, который настоящая политика отвергает.

Действующая редакция **ГОСТ Р 52872-2019** этого **не требует**: она формулирует требования к доступности **самого интерфейса** на уровне AA, а не к существованию альтернативной версии. Подход новой редакции опирается на WCAG и согласован с международной инклюзивной практикой.

**Однако инерция надзорной практики реальна.** Аккредитационные эксперты, прокурорские проверки и часть надзорных органов по привычке могут искать на сайте «кнопку для слабовидящих» и трактовать её отсутствие как нарушение.

**Каждый продукт экосистемы обязан** содержать в публичной декларации доступности (см. §7.5) явный раздел **«Почему отсутствует отдельная версия для слабовидящих»**. Этот раздел:
- ссылается на действующую редакцию ГОСТ Р 52872-2019 и её отличие от редакции 2012 года;
- разъясняет архитектурное и инклюзивное обоснование единого доступного интерфейса;
- указывает, что соответствие AA обеспечивает доступность для слабовидящих в рамках единой кодовой базы.

Это переводит позицию из «у нас нет — мы так решили» в «у нас нет — закон этого не требует, а архитектура полнее обеспечивает доступность». Проверяющий получает ответ до того, как выпишет замечание.

### 1.3. Доступность ≠ функция доступности

Опции типа «крупный шрифт», «высокий контраст», «озвучивание» — это **настройки пользователя**, реализуемые поверх доступной базовой платформы. Они не заменяют доступности, а используют её.

**Опасный антипаттерн** — считать наличие кнопки «версия для слабовидящих» проявлением доступности. Это маркер того, что базовая платформа недоступна.

## 2. Слои доступности

Доступность распространяется не только на UI, но на **все слои** системы.

### 2.1. Интерфейс (UI)

Семантический HTML, ARIA, клавиатурная навигация, контраст, фокус, тап-цели, альтернативы для нетекстового контента. Это **наиболее видимый слой**, но не единственный.

Регулируется техническими документами продуктов (например, `didakticon-accessibility.md`).

### 2.2. Данные

> **Данные должны быть семантичными.** Атрибут сущности фиксирует **смысл**, а не визуальное представление.

**Антипаттерн:**
```
event.status_color = "red"
event.priority_color = "red"
event.indicator = "🔴"
```

**Корректно:**
```
event.status = "cancelled"
event.priority = "critical"
event.indicator_kind = "warning"
```

Цвет и визуальный значок выбираются **на стороне рендера** из семантического статуса. Это даёт:
- единое поведение во всех каналах представления (UI, печатный документ, голосовой помощник, экспорт);
- возможность скринридеру прочитать «отменено», а не «красное»;
- независимость от темы (в тёмной теме «отменено» может быть другим цветом).

Регулируется на уровне доменных спецификаций edu-framework.

### 2.3. API

> **API не передаёт смысл через визуальные признаки.**

Контракты JSON-RPC методов, REST-эндпоинтов и event-сообщений между сервисами оперируют семантическими полями. Визуальное представление — забота клиента.

Применимо к: Универкон-API, Тестикон-протокол, manifest пакета урока, контракты адресных коммуникаций, событиям Хроникона.

Это критично, потому что **через API экосистема расширяется** на новые каналы: Telegram-боты, мобильные приложения, голосовые помощники, чат-боты для абитуриентов. Если API передаёт `priorityColor=red`, голосовой помощник не сможет сообщить «критический приоритет» — потому что для него «red» бессмысленен.

### 2.4. Контент (педагогический)

Материалы пакетов уроков, лекции, тесты, задания. Доступность контента — **зона ответственности автора** (педагога), но платформа **обязана**:
- **помогать** автору создавать доступный контент (предупреждения о пропущенных alt, генерация субтитров, проверка читабельности);
- **декларировать** уровень доступности каждого артефакта (см. §4 в `edu-framework-domain-merge-map.md`);
- **измерять** доступность контента ОО (метрика Accessibility Score — определение и защита от Goodhart-эффекта будут зафиксированы в следующей итерации политики).

Регулируется доменом «Пакеты уроков» edu-framework.

#### 2.4.1. Импортируемый сторонний контент

Для контента, импортируемого из внешних источников (SCORM-пакеты сторонних вендоров, видеоэмбеды YouTube/МТС Линк, OER-материалы Open edX), платформа физически не может гарантировать доступность вложенного содержимого.

**Применяемые правила:**
- Платформа гарантирует доступность **обёртки-плеера** (фокус, клавиатура, ARIA-обрамление iframe).
- Декларация доступности самого импортируемого артефакта может содержать значение **`unknown`** — это **честная декларация**, отличная от запрещённого «по умолчанию unknown» для собственного контента.
- Декларация платформы при представлении импортируемого артефакта явно сообщает пользователю: «доступность вложенного контента не гарантирована платформой».
- Если импортируемый материал критичен для освоения дисциплины и при этом недоступен — применяется механизм запроса альтернативного формата (см. соответствующие продуктовые спецификации).

Это устраняет иначе неразрешимую коллизию: запрет приёмки недоступного (см. §4.2) применительно к материалу, который ОО обязана выложить и доступность которого вне нашего контроля.

### 2.5. Документы (генерируемые)

PDF, docx, xlsx, выгрузки, печатные ведомости, диплом, дипломные приложения, договоры, счета.

> **Tagged PDF, корректный reading order, текстовые альтернативы — обязательное требование** к любым документам, генерируемым системами экосистемы.

Это часто **главный провал** инклюзии: ЭИОС в UI доступна, а скачанная ведомость — картинка без структуры. Аккредитация проверяет именно конечный артефакт.

Регулируется на уровне рендер-движков (Универкон-PDF, Кабиторий-docx, и т.д.). Должно быть отдельной проекцией каноничной модели документа.

## 3. Единая точка реализации — Design System

### 3.1. Принцип

> **Все требования доступности интерфейса реализуются один раз — в дизайн-системе экосистемы. Прикладной разработчик не пишет ARIA вручную.**

Разработчик использует компоненты:

```
Button, Input, Checkbox, Radio, Select, Toggle, Slider,
Dialog, Sheet, Drawer, Popover, Tooltip,
Tabs, Accordion, Stepper, Wizard,
Table, DataGrid, Calendar, DatePicker,
Toast, Banner, Alert,
NavigationBar, SideBar, BottomTabs, TopBar
↓
доступность встроена
```

Каждый компонент дизайн-системы:
- проходит тесты со скринридерами;
- покрыт `axe` и Storybook a11y addon;
- имеет ARIA-разметку, корректный focus management, клавиатурную навигацию;
- декларирует свои гарантии доступности в документации.

### 3.2. Что попадает в дизайн-систему

В дизайн-систему попадает **всё повторяющееся**. Если компонент используется в двух местах — он не делается дважды.

Это особенно важно для составных компонентов, где доступность сложна:
- модалки и шторки (focus trap, return focus, Esc, фокусированный заголовок);
- табы и аккордеоны (ARIA-роли, навигация стрелками);
- комбобоксы и автокомплиты (объявление вариантов скринридером);
- таблицы данных (`<th scope>`, `<caption>`, режим таблицы);
- drag-and-drop с клавиатурной альтернативой.

### 3.3. Запрет на ad-hoc компоненты

> **Прикладной код не имеет права реализовывать стандартный UI-паттерн заново.** Если в дизайн-системе компонента нет — он сначала добавляется в дизайн-систему, потом используется в прикладном коде. Не наоборот.

Это защищает от:
- copy-paste с потерей доступности;
- расхождения между экранами по UX;
- невозможности централизованно обновить ARIA при выходе новой версии WCAG.

### 3.4. Технологический выбор

Для React-стека экосистемы канонические основы:
- **Radix UI** или **Headless UI** — для базовых ARIA-корректных примитивов.
- Поверх — стилизация под бренд Univerkon (Tailwind с CSS-переменными).

**Запрещено:** катать «свою библиотеку компонентов с нуля» без явного архитектурного решения. Это технический долг доступности с момента появления.

## 4. Accessibility Debt

### 4.1. Определение

> **Accessibility Debt** — любой принятый в систему артефакт (компонент, экран, спецификация, API-контракт, документ), не удовлетворяющий требованиям доступности своего слоя на момент приёмки.

### 4.2. Запрет приёмки

> **Запрещается принимать в систему компонент, требующий последующей адаптации под доступность.**

Конкретно:
- Pull request с нарушением a11y-лимита не сливается, даже если функционально работает.
- Доменная спецификация без атрибутов доступности (где применимо) не вливается в edu-framework.
- API-контракт с визуальными признаками вместо семантических не принимается в публичный API.
- Пакет урока без декларации доступности не публикуется. Декларация может содержать `not_applicable` для конкретных пунктов (с обоснованием), а для импортируемого стороннего контента — `unknown` (см. §2.4.1). «По умолчанию unknown» для собственного контента — запрещено.
- Документ без tagged-структуры не отдаётся как итоговый артефакт ЭИОС (черновики — можно, итоговые ведомости — нет).

### 4.3. Классификация долга

Не всякий Accessibility Debt равноценен. Для приоритизации и отчётности применяется трёхуровневая классификация:

**Critical** — нарушение **полностью блокирует** использование функции хотя бы одной категорией пользователей. Например: форма отправки ответа на тест не работает скринридером; ведомость в PDF не имеет тегов и не читается скринридером вовсе.
- **SLA:** хотфикс. Эскалируется как функциональный блокер.
- Обходной путь, если он есть, оформляется как временная мера на странице `/accessibility`.

**Major** — функция работает, но **существенно затрудняет** доступ. Обходной путь существует и достижим. Например: drag-and-drop без клавиатурной альтернативы при наличии возможности использовать выпадающий список через настройки.
- **SLA:** включается в ближайший плановый релиз (типично 2–4 недели).

**Minor** — нарушение рекомендаций без блокировки функциональности. Например: контраст вторичной подписи на грани AA, отсутствие `aria-label` у некритичной иконки.
- **SLA:** ближайший спринт работ по техническому долгу, не блокирует релизы.

### 4.4. Учёт долга

Каждый зарегистрированный Accessibility Debt содержит:
- класс тяжести (Critical / Major / Minor);
- что нарушено (с указанием слоя и конкретного критерия);
- кем принят и в рамках какого артефакта;
- срок исправления;
- временные меры для пользователей;
- ссылку на задачу/тикет, в котором ведётся работа по исправлению.

Долг видим, измерим, классифицирован и регулярно сокращается. Скрытый долг — недопустим.

Реестр ведётся **архитектором доступности** (см. §6).

## 5. Definition of Done

> **Задача считается выполненной только при соответствии чеклисту доступности.**

Применимо к каждому PR/задаче в любом продукте экосистемы. Конкретный чеклист — артефакт каждой команды/продукта.

### 5.1. Минимальный UI-чеклист

Минимально необходимый набор для **UI-уровня** (слой 2.1):

- [ ] Проходит `axe` без нарушений уровня serious / critical.
- [ ] Все интерактивные элементы достижимы клавиатурой (Tab, Esc, Enter, Space, стрелки).
- [ ] Видимый focus-индикатор на всех фокусируемых элементах.
- [ ] Не сломан при увеличении до 200% и при `prefers-reduced-motion`.
- [ ] Контраст соответствует §7.6 в обеих темах.
- [ ] У всех `<img>` есть `alt` (декоративные — `alt=""` явно).
- [ ] У всех полей форм есть `<label>` или `aria-label`.
- [ ] Иконочные кнопки имеют `aria-label`.
- [ ] Сообщения об ошибках связаны с полями через `aria-describedby`.
- [ ] Динамические изменения объявляются через `aria-live` (где применимо).
- [ ] Используются компоненты дизайн-системы, а не самописные эквиваленты (или есть архитектурное обоснование).

Этот чеклист **необсуждаем**. PR без него считается незавершённым.

### 5.2. Per-layer чеклисты — TODO

Для других слоёв доступности (§2.2 данные, §2.3 API, §2.5 документы, §2.4 контент) **аналогичные per-layer чеклисты** требуются, но в текущей версии политики **не разработаны**.

Это известный недочёт. Пока запрет приёмки из §4.2 для не-UI слоёв держится на ручном ревью архитектора фреймворка и архитектора доступности. Полноценный per-layer DoD будет введён в следующей итерации политики вместе с автоматизированными гейтами (линтер OpenAPI-схем, валидатор PDF/UA).

## 6. Матрица ответственности

| Уровень | Кто отвечает | За что отвечает |
|---|---|---|
| **Платформа (edu-framework)** | Архитекторы фреймворка | Доменные модели с семантическими атрибутами, контракты API без визуальных признаков, паттерны фреймворка (видимость, view-by-role, accessibility declaration). |
| **Доступность как сквозная функция** | Архитектор доступности (a11y owner) | Ведение реестра Accessibility Debt; периодическая ревизия сроков и эскалация просрочки; обновление политики при изменении WCAG/ГОСТ; координация ревью между уровнями; ведение нормативного реестра числовых порогов (§7.6); ответственность за процесс переоценки публичных деклараций. Эта роль может совмещаться с архитектором фреймворка, но обозначается явно. |
| **Дизайн-система** | Команда дизайн-системы | ARIA-корректные компоненты, тесты со скринридерами, документация гарантий, типографические токены, токены контраста, конкретные per-component чеклисты a11y. |
| **Продукт (Дидактикон, Кабиторий, и т.д.)** | Команда продукта | Использование компонентов дизайн-системы, корректная композиция, SPA-маршрутизация с управлением фокусом, локализация (`lang`), декларация соответствия. |
| **Контент (педагогический)** | Автор курса (педагог) | Субтитры, alt-тексты, расшифровки, простой язык, корректная структура учебного материала, заполненная декларация доступности пакета. |
| **Документы (рендер)** | Рендер-движки продуктов | Tagged PDF, корректный reading order, метаданные документа, альтернативы для нетекстового контента. |
| **Организация (вуз, ОО)** | Администрация ОО | Физическая среда (пандусы, лифты), сурдоперевод видеолекций, адаптированные образовательные программы (АОПОП), обучение педагогов созданию доступного контента, центр инклюзивного образования, координация с тестированием. |
| **Студент / пользователь** | Сам пользователь | Сообщение системе своих предпочтений интерфейса (`accessibility_preferences`), обратная связь о проблемах доступности. |

Эта матрица **снимает споры «кто виноват»** при инцидентах доступности и определяет, кому адресуется конкретный пункт обнаруженного нарушения.

## 7. Соответствие стандартам

### 7.1. Базовый стандарт

> **ГОСТ Р 52872-2019**, уровень соответствия **AA**.
> ГОСТ опирается на WCAG 2.1. Все критерии AA уровня WCAG 2.1 — обязательны.

### 7.2. WCAG 2.2 — выборочно

Критерии WCAG 2.2, появившиеся после ГОСТ Р 52872-2019 и **включаемые** в политику экосистемы Univerkon:
- **2.4.11 Focus Not Obscured** (AA) — фокус не должен уезжать под sticky-элементы.
- **3.3.8 Accessible Authentication** (AA) — не блокировать paste в пароль, поддерживать passkeys, не использовать когнитивные CAPTCHA без альтернативы.

Эти критерии формально **не предписаны** действующим ГОСТом, но включаются, потому что:
- бьют ровно по специфичным для PWA / mobile-first паттернам;
- дёшевы в реализации;
- с высокой вероятностью войдут в следующую редакцию ГОСТ;
- ожидаемы аккредитационной комиссией в её актуальном понимании доступности.

Критерий **2.5.8 Target Size Minimum** (AA, 24×24) — формально из WCAG 2.2 — **не вносится отдельным пунктом**, поскольку выполняется автоматически как следствие более строгого критерия 2.5.5 (см. §7.3).

### 7.3. WCAG 2.2 AAA — где получается без боли

AAA не обязателен. Конкретные пункты AAA, которые **берутся в политику** экосистемы:
- **2.5.5 Target Size 44×44** — берётся как более строгая версия требования к размеру тап-цели; критерий 2.5.8 (24×24, AA) выполняется как следствие.
- **2.4.8 Location** — навигационные крошки/индикация места.
- **1.4.6 Contrast Enhanced (7:1)** — для основного текста (рекомендация, не обязательство).

### 7.4. Российская нормативная база

- **ФЗ-273 «Об образовании в РФ», ст. 79** — обязывает обеспечить условия обучения для лиц с ОВЗ.
- **Приказы Минобрнауки об аккредитации** — учитывают доступность ЭИОС как проверяемый показатель.
- **152-ФЗ** — определяет режим работы со специальными категориями ПДн (см. §8).

### 7.5. Декларация соответствия

> Каждый продукт экосистемы, имеющий пользовательский интерфейс, обязан публиковать **декларацию доступности** — публичный документ, оформленный в духе WCAG conformance claim и адаптированный под требования ГОСТ Р 52872-2019.

Декларация доступна по стандартизованному URL **`/accessibility`** относительно домена продукта.

**Обязательное содержание декларации:**
- уровень соответствия (AA);
- перечень разделов ГОСТ Р 52872-2019, по которым достигнуто соответствие;
- известные ограничения (если есть);
- раздел **«Почему отсутствует отдельная версия для слабовидящих»** (см. §1.2.1);
- канал обратной связи о проблемах доступности (форма, email);
- дата последней оценки;
- дата следующей плановой переоценки.

Декларация — **формальная точка проверки** аккредитационной комиссией и реальная точка обратной связи для пользователей.

**Процесс переоценки декларации:** даты, периодичность, ответственный — задача архитектора доступности (см. §6). Полноценный регламент переоценки и процедура внешнего аудита — в следующей итерации политики.

### 7.6. Нормативный реестр числовых порогов (Single Source of Truth)

> **Все числовые пороги доступности задаются единожды в этом разделе.** Продуктовые документы и спецификации дизайн-системы **ссылаются** на пороги, но не дублируют их.

При ревизии WCAG / ГОСТ / внутренних решений — пороги обновляются **только здесь**. Это устраняет рассинхрон при будущих изменениях.

**Контраст:**
- Основной текст: ≥ 4.5:1 (AA);
- Крупный текст (≥ 18pt или ≥ 14pt жирный): ≥ 3:1 (AA);
- Границы интерактивных элементов: ≥ 3:1;
- Усиленный контраст основного текста (рекомендация): ≥ 7:1 (AAA).

**Размер тап-цели:**
- Минимальный: 44 × 44 CSS-пикселя (берётся 2.5.5 AAA как более строгая версия 2.5.8 AA);
- Расстояние между интерактивными элементами: ≥ 8 px.

**Типографика:**
- Базовый размер шрифта: ≥ 16 px (= 1 rem);
- Межстрочный интервал длинного текста: ≥ 1.5;
- Минимальная ширина viewport без горизонтального скролла: 320 CSS-пикселей.

**Мигание и анимация:**
- Максимум вспышек в секунду: ≤ 3.

**Ответственный за реестр:** архитектор доступности (см. §6). Изменения вносятся как полноценная ревизия политики, не точечно.

## 8. Юридические границы

### 8.1. Специальные ПДн — не хранятся в учебных системах

> **Системы экосистемы Univerkon не хранят сведения о состоянии здоровья, инвалидности, диагнозах пользователей.** Эти данные относятся к специальным категориям ПДн (ст. 10 152-ФЗ) и требуют отдельной правовой основы, явного согласия и режима повышенной защиты.

Если ОО ведёт учёт таких сведений (для предоставления льгот, академического отпуска по медицинским показаниям) — это происходит **вне** учебных систем, в кадровом или социальном контуре, со своим режимом защиты.

### 8.2. accessibility_preferences ≠ медицинские данные

> Поле `accessibility_preferences` физлица (см. `edu-framework-domain-merge-map.md`, §7.1) содержит **только саморепортируемые предпочтения интерфейса**: размер шрифта, контраст, использование скринридера, чувствительность к движению.

Это **не медицинские данные**:
- не зависит от диагноза (зрячий человек может предпочитать крупный шрифт);
- декларируется самим пользователем без подтверждения;
- не используется для каких-либо решений, кроме рендера интерфейса.

Эта формулировка фиксируется явно, чтобы не запускать процедуры специальных ПДн там, где они не нужны, и не блокировать функциональность из-за избыточных требований защиты.

### 8.3. Запрос альтернативного формата

Студент имеет право запросить альтернативный формат недоступного материала через канал обратной связи продукта. Запрос направляется:
- автору контента (педагогу) — если речь о пакете урока;
- администрации ОО — если речь о системном документе;
- команде продукта — если речь о баге доступности.

Сам факт запроса **не раскрывает причину** (необязательно указывать диагноз). Достаточно описания, какой формат нужен.

## 10. Эволюция политики

### 10.1. Версионирование

Политика версионируется как часть edu-framework. Минорные обновления — без обратной несовместимости. Мажорные — с переходным периодом и явной картой миграции.

### 10.2. Источники изменений

- Обновления WCAG (2.2, потенциально 3.0).
- Обновления ГОСТ Р 52872 (ожидается ревизия).
- Изменения 152-ФЗ и сопутствующих регламентов.
- Опыт эксплуатации (мониторинг использования a11y-фич, обратная связь, инциденты).
- Решения архитектора доступности и архитектора фреймворка.

### 10.3. Принцип

> Политика **не смягчается** под давлением сроков, бюджета или удобства разработки. Она может **расширяться** в сторону строгости или **уточняться** в сторону ясности, но не отменяться.

Любое временное послабление оформляется как зарегистрированный Accessibility Debt с явным сроком исправления.

### 10.4. План следующей итерации (v3 — структурный сдвиг)

В следующей итерации политики будут зафиксированы:
- доступность как **cross-cutting concern**, пересекающая все домены экосистемы (новая рамка восприятия документа);
- **Non-goals** — явное определение того, чего политика не описывает;
- **Архитектурные аксиомы** — компактные цитируемые формулировки;
- **Принцип деградации** — альтернативные пути выполнения для функций, использующих недоступные технологии;
- **Offline-доступность** как доступность образовательного процесса (PWA-специфика);
- **Метрики и зрелость** — определение Accessibility Score с защитой от Goodhart-эффекта, метрики продукта;
- **Legacy и миграция** — план работы с уже загруженным контентом;
- **Обучение и компетенции** — обязательность вводного курса по доступности;
- **Машиночитаемость и AI-готовность** — сохранение смысла при представлении через LLM-агенты;
- **Per-layer Definition of Done** для всех слоёв доступности (§5.2);
- **Автоматический enforcement не-UI слоёв** — линтер OpenAPI-схем, валидатор PDF/UA;
- **Расширенная матрица ответственности** — DevOps, служба поддержки, служба обучения;
- **Регламент внешнего аудита** и периодичность переоценки декларации.
