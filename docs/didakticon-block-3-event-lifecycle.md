**Статус:** в работе, продолжающаяся фиксация

**Дата:** 30 июня 2026
**Контекст:** экосистема Univerkon, edu-framework

## Как читать этот документ

Документ описывает третий блок Дидактикона — карточку учебного события
как общую рабочую поверхность для студента и педагога.

В версии 0.3 добавлены: педагогическая проекция карточки события (4
зоны), runtime-пульт педагога в `идёт` (3 зоны), классификация форм по
`time_model` (inline / extended), защита от переработки занятия,
принудительное завершение inline-форм при `завершено`, два жанра
бонусных форм (pre_event и post_event) с pickup-механикой, принцип
«один главный педагог на занятии».

Не описано в этой версии: специальные форматы (онлайн через ВКС,
гибрид, день практики, итоговая аттестация), технический контракт
outbox с Универконом, многопедагогические сценарии (вне MVP).
Подсистема подтверждения присутствия (presence verification) — в
поперечный документ.

Документ — рабочая спецификация. Архитектурные решения этой версии
зафиксированы.

## 1. Назначение блока

Блок III решает четыре задачи:

1. Описать жизненный цикл учебного события.
2. Определить, какие факты возникают в каждой фазе и куда они уходят.
3. Дать модель runtime внутри состояний `регистрация` и `идёт`:
   видимость материалов, классы форм контроля, режимы запуска,
   журнализация попыток.
4. Дать общую модель карточки события — одна сущность, две проекции
   (студент и педагог).

---

## 2. Несущие принципы

1. **Карточка события — одна сущность, две проекции.**
2. **Событие из УП не отменяется, только переносится.**
3. **Любой перенос — два факта:** срыв исходного окна и появление
   нового. Атомарный ChangeSet в расписании.
4. **Любой перенос требует объяснения от педагога.** Объяснительная —
   кадровый документ в **ЛК сотрудника**, не в Дидактиконе.
   Уважительность — акт ОО постфактум.
5. **Дистанционно начать занятие нельзя.**
6. **Сроки обязательств фиксируются в момент `завершено`** и дальше
   неизменяемы.
7. **Двухфазность форм контроля.**
8. **Принятие — бинарное.** Отрицательная оценка рождает обязательство-
   отработку. Никакого цикла «доработка той же попытки».
9. **Регистрация — процесс аутентификации присутствия.**
10. **Пакет события — банк допустимых форм, не план.**
11. **Видимость материалов нормативно обусловлена.** ФОС всегда
    `enrolled`.
12. **Один главный педагог на занятии.** Никакого разделения
    ответственности. Многопедагогические сценарии — вне MVP.
13. **Inline-формы укладываются в событие.** Защита от опоздания
    студента на следующее занятие.

---

## 3. Терминология

**Учебное событие** — генератор обязательств. §4 академической концепции.

**Карточка события** — поверхность Дидактикона с метаданными,
состоянием, действиями.

**Состояние события** — фаза жизненного цикла. Шесть состояний (§4).

**Педагогическая проекция карточки события** — ракурс для роли
`instructor` с четырьмя зонами (§16).

**Runtime-пульт** — поверхность педагогической проекции в состоянии
`идёт`, где живут запуск форм, индивидуальные вызовы, мониторинг
(§17).

**Регистрация** — процесс подтверждения присутствия студента.

**Видимость материала** — `visibility_gate`.

**time_model формы контроля** — `inline` (с `duration_minutes`, в
пределах события) или `extended` (с `due_at` / `delivery_due_at`,
переживает событие).

**Cohort** — аудитория запуска формы: `all` или `single`.

**Assignment group** — группа взаимозаменяемых вариативных форм.

**Pickup** — акт студента «взять pre_event-бонус» (§18).

**Pre_event-бонус** — бонусная форма, доступная студенту **до**
события, для подготовки к нему.

**Post_event-бонус** — бонусная форма, доступная студенту **после**
события, для повышения рейтинга самостоятельно.

**Пять фактов формы контроля** — assignment → launch → attempt →
submission → evaluation. Для pre_event-бонусов добавляется первый
факт `pickup`.

## 4. Состояния учебного события

```
запланировано → регистрация → регистрация_закрыта → идёт → завершено

   (из запланировано, регистрация или регистрация_закрыта по таймауту)
                  ↘
              перенесено (rescheduling × explanation)
```

### 4.1. запланировано

Событие создано расписанием. До открытия регистрации — далеко.

### 4.2. регистрация

Идёт процесс подтверждения присутствия. Триггер открытия — N минут до
плана (политика) или освобождение ресурса-локации. Закрытие — в момент
регламентного начала по плану.

### 4.3. регистрация_закрыта

Регистрация прекращена, занятие ещё не начато. Состояние фиксирует
факт длительности задержки педагога.

### 4.4. идёт

Основная часть события. Триггер — действие педагога «начать» (требует
подтверждения присутствия) или таймаут 15 минут (автоперенос).

### 4.5. завершено

Карточка архивная. Материалы read-only. Inline-формы **принудительно
завершаются** (§ 11.6). Extended-формы продолжают жить со своими
дедлайнами. Post_event-бонусы становятся доступны студенту для
самостоятельного запуска.

### 4.6. перенесено

Два независимых подпроцесса:

- `rescheduling`: pending → scheduled (диспетчер подбирает новое окно).
- `explanation`: pending → submitted → reviewed_acceptable / reviewed_unacceptable.

**Подача объяснительной — не в Дидактиконе.** Это кадровый документ,
живёт в ЛК сотрудника. Универкон при переходе в `перенесено(pending)`
отправляет педагогу уведомление через критический канал
(`external_action` с deep-link в ЛК). Результат рассмотрения возвращается
как прикладное уведомление в инбокс Дидактикона.

Дидактикон закрывает карточку события из активного списка, когда
**оба** подпроцесса завершены: `rescheduling: scheduled` И
`explanation` в одном из reviewed-состояний.

## 5. Подтверждение присутствия (presence verification)

Подсистема выделена в поперечный документ

Архитектурная природа — частный случай аутентификации с

Подсистема общая для подтверждения присутствия педагога (переход в
`идёт`) и регистрации студента.

## 6. Запреты

1. **Дистанционно начать очное занятие нельзя.**
2. **Событие из УП нельзя отменить.** Только перенести.
3. **Дедлайн обязательства не редактируется после `завершено`.**
4. **ФОС не может быть скрыт от зачисленного студента.**
5. **На занятии может быть только один главный педагог.** Никакого
   разделения runtime-действий.

---

## 7. Сроки обязательств — двухслойная фиксация

Стек настроек, разрешение в timestamp в момент `завершено`:

```
1. Дефолт по типу формы контроля       (ОО)
2. Учётная политика подразделения       (переопределяет)
3. РПД дисциплины (base пакета)         (переопределяет)
4. Overlay педагога в пакете            (переопределяет)
5. Runtime в момент `завершено`         (разрешение)
```

У обязательства два срока: `readiness_due_at` (готовность) и
`delivery_due_at` (сдача).

---

## 8. Двухфазность форм контроля и бинарное принятие

```
открыто → готово → сдано → оценено
                            ├─ положительно → закрыто
                            └─ отрицательно → рождает обязательство
                                              «отработка»
```

Каждая попытка — отдельная запись. Никакого цикла «доработай ту же
попытку».

**Апелляция** — отдельная процедура от подачи (студент жмёт
«Обжаловать» на карточке результата) до рассмотрения senior_grader.
Подача и рассмотрение целиком в Дидактиконе (учебный процесс, не
кадровый). Корректирующая запись по §5 академической концепции.

## 9. Видимость материалов пакета

### 9.1. Четыре уровня

| Уровень | Когда виден | Примеры |
|---|---|---|
| `enrolled` | с зачисления на дисциплину | программа, ФОС, банк вопросов, темы письменных работ, критерии, шкалы, литература |
| `pre_event` | с открытия события | материалы для подготовки, чтения, overlay педагога |
| `during_event` | после регистрации в событии | слайды, текст контрольной, конспект |
| `teacher_unlocked` | явным действием педагога | разборы решений, доп. материалы |

### 9.2. Нормативные ограничения

ФОС — всегда `enrolled`. Конкретная выборка вопросов в попытке — не
часть видимости пакета, а семантика рандомизации Тестикона.

### 9.3. Материалы дисциплины vs материалы пакета

```
Дисциплина (РПД)
  ├── материалы дисциплины (visibility_gate = enrolled)
  └── Пакет занятия N
        ├── материалы пакета (pre_event / during_event / teacher_unlocked)
        └── формы контроля события
```

## 10. Gate-логика входа в работу

Матрица доступа (состояние × факт регистрации):

| Состояние | Регистрация | Материалы | Формы контроля |
|---|---|---|---|
| `запланировано` | — | enrolled + pre_event | нет |
| `регистрация` | нет | enrolled + pre_event | нет |
| `регистрация` | отмечен | enrolled + pre_event + during_event | нет |
| `регистрация_закрыта` | отмечен | enrolled + pre_event + during_event | нет |
| `регистрация_закрыта` | не отмечен | enrolled + pre_event | нет |
| `идёт` | отмечен | enrolled + pre_event + during_event (+ teacher_unlocked) | активные запущенные |
| `идёт` | не отмечен | enrolled + pre_event | нет |
| `завершено` | отмечен | enrolled + pre_event + during_event (read-only) | завершённые + post_event-бонусы |
| `завершено` | не отмечен | enrolled + pre_event | нет |
| `перенесено` | — | enrolled | нет |

Конфликт «прошёл задания, но педагог снял отметку» — арбитраж
деканатом, не системой.

## 11. Классы форм контроля

### 11.1. role_in_rating

- `required` — обязательная, входит в проходной порог.
- `bonus` — повышающая, добровольная.

### 11.2. assignment_mode

- `all` — назначается всем зарегистрированным.
- `per_student` — назначается каждому индивидуально.

### 11.3. time_model

- `inline` — должна уложиться в событие. Имеет `duration_minutes`.
- `extended` — переживает событие. Имеет `due_at` / `delivery_due_at`.

Взаимоисключающие. Форма имеет одну `time_model`.

### 11.4. Три практических класса

| Класс | role_in_rating | assignment_mode | Кто запускает | Когда |
|---|---|---|---|---|
| **Обязательная** | required | all | педагог | в `идёт` |
| **Вариативная** | required | per_student | педагог | в `идёт` |
| **Бонусная (post_event)** | bonus | all | студент | в `завершено` |
| **Бонусная (pre_event)** | bonus | all | студент | до события (после pickup) |

### 11.5. Assignment group — вариативные

Группа взаимозаменяемых вариантов. Назначение одной из группы
конкретному студенту автоматически снимает остальные для него.

### 11.6. Inline-формы — принудительное завершение

Inline-форма имеет `duration_minutes` (например, 30 минут). Запускается
в `идёт`, должна уложиться **в пределах события**.

**Защита от переработки.** Перед запуском педагог в runtime-пульте
видит запас времени:

```
сколько_осталось_события = planned_end_at - now
сколько_надо_форме       = duration_minutes
запас                    = сколько_осталось_события - сколько_надо_форме
```

Если запас отрицательный — кнопка запуска **требует осознанного
подтверждения** «запустить под ответственность». Запуск с
переработкой журналируется:

```
launch:
  launched_with_overrun: true
  expected_overrun_minutes: 5
```

Это даёт аналитику ОО «педагоги, систематически запускающие
inline-формы с переработкой».

**Принудительное завершение.** В момент перехода события в
`завершено` все запущенные inline-формы **принудительно завершаются**.
Студенты, не успевшие сдать, получают `expired_attempt`.

Это охранник от ситуации «студент доходит тест в коридоре, пока идёт
на следующую пару». Inline по природе не может пережить событие.

### 11.7. Extended-формы — переживают событие

Extended-форма имеет `due_at` / `delivery_due_at`, не `duration_minutes`.
Запускается в `идёт`, но **не имеет рантайм-таймера в моменте**. Это
обязательство, выданное на занятии, со сроком готовности по правилам §7.

При переходе события в `завершено` extended-формы продолжают жить.
Студент сдаёт в свой срок.

Если у студента активная extended-попытка — на дашборде студента
карточка `active_attempt` (§ блока II).

### 11.8. Бонусы — два жанра

#### 11.8.1. Post_event-бонус (`availability_window: post_event`)

Доступен студенту **после** перехода события в `завершено`. Запускает
сам студент. Цель — повышение рейтинга. Атрибуты:

```yaml
- id: fc-bonus-post
  role_in_rating: bonus
  assignment_mode: all
  availability_window: post_event
  time_model: extended
  delivery_due_offset_days: 7
```

#### 11.8.2. Pre_event-бонус (`availability_window: pre_event`)

Доступен студенту **до** события. Студент **берёт** бонус (pickup),
готовится дома, результат представляет на занятии. Атрибуты:

```yaml
- id: fc-bonus-pre
  role_in_rating: bonus
  assignment_mode: all
  availability_window: pre_event
  pickup_deadline_offset_hours: 24    # за 24 часа до плана
  presentation_required: true          # требует устного представления
```

Pre_event-бонусы появляются:
- На дашборде студента — индикатор `has_pre_event_bonus: true` в
  карточке `event`.
- В расписании студента — подсветка занятий с доступными
  pre_event-бонусами (настройкой студента).
- В банке форм runtime-пульта педагога — отдельной секцией
  «Подготовленное от студентов», с указанием кто взял и кто готов.

Бонусы по-прежнему **не порождают долгов**. Если студент взял, но не
подготовил — никакого штрафа. Если никто не взял — занятие идёт по
штатному сценарию пакета.

## 12. Запуск форм контроля

### 12.1. Два режима — одна механика

`cohort: all` — групповой. `cohort: single(student_context_id)` —
индивидуальный вызов («к доске Иванов»). Журналируется один и тот же
факт `launch` с разным `cohort`.

### 12.2. Кто и когда запускает

| Класс | Кто | Когда |
|---|---|---|
| Обязательная (`all`) | педагог | в `идёт`, в нужной точке |
| Вариативная (`per_student`) | педагог | в `идёт` |
| Post_event-бонус | студент | в `завершено` |
| Pre_event-бонус | студент берёт (pickup) до события, представляет на занятии; педагог запускает в `идёт` | в `идёт` (после pickup) |

### 12.3. Пакет как банк, не план

Пакет описывает **возможности**. То, что не запущено — не порождает
обязательств. Исключения:

- **Обязательная** форма не запущена → долг педагога
  (`teacher_event_debt`).
- У студента нет ни одной назначенной из обязательной
  `assignment_group` к `завершено` → долг педагога перед этим
  студентом.

## 13. Жизненный цикл формы контроля — пять (или шесть) фактов

Для обычных форм:

```
assignment → launch → attempt → submission → evaluation
```

Для pre_event-бонусов добавляется первый факт:

```
pickup → assignment → launch → attempt → submission → evaluation
```

### 13.1. Поля

```
pickup:     event_id, form_control_id, student_context_id,
            picked_up_at

assignment: event_id, student_context_id, form_control_id,
            assignment_group?, assigned_by, assigned_at,
            assignment_basis (explicit | bulk_apply | from_pickup)

launch:     event_id, form_control_id, cohort,
            launched_by, launched_at,
            launched_with_overrun: bool,        # для inline
            expected_overrun_minutes: int?      # для inline

attempt:    event_id, form_control_id, student_context_id,
            attempt_id, started_at,
            fos_snapshot_id,
            deadline_at  (для inline = launched_at + duration_minutes;
                         для extended = launched_at + extended-срок)

submission: attempt_id, submitted_at, submission_kind,
            content_ref

evaluation: attempt_id, evaluated_by, evaluated_at,
            grade, grade_is_positive, comment?
```

### 13.2. Принудительное завершение inline

Для inline-формы в момент `завершено` события для всех студентов с
неактивной submission генерируется автоматическое `submission` со
статусом `expired` и далее `evaluation` со статусом `expired_no_attempt`
или `expired_during_attempt`.

### 13.3. Связь с моделью обязательств

Пять фактов — жизненный цикл одной попытки. Отрицательная оценка
рождает новое обязательство (отработка) → новые пять фактов на новой
попытке.

## 14. Педагогический бонус — отдельный механизм

Не из пакета и не из ФОС. Педагог в любой момент даёт студенту баллы
с указанием причины. Лимит — политика подразделения.

Не относится к жизненному циклу формы контроля. Самостоятельный факт:

```
pedagogical_bonus:
  student_context_id, discipline_id, module_id,
  points, reason, awarded_by, awarded_at
```

Прозрачен студенту. Учитывается в БРС наравне с пакетными формами.

## 15. Что фиксируется в Универконе

**События верхнего уровня:**

| Поле | Когда |
|---|---|
| `event.id` | создание |
| `event.planned_start_at`, `planned_end_at` | создание |
| `event.registration_opened_at` | переход в `регистрация` |
| `event.actually_started_at` | переход в `идёт` |
| `event.actually_ended_at` | переход в `завершено` |
| `event.postponed_at`, `postponement_reason_code` | переход в `перенесено` |
| `event.replacement_event_id` | подбор нового окна |
| `event.explanation_id`, `submitted_at`, `verdict`, `verdict_at` | пакет уважительности (из ЛК) |
| `event.delay_seconds` | вычисляемое |

**Регистрация студента:**

```
attendance.registered_at
attendance.factors_used
attendance.status (pending | confirmed | rejected)
attendance.validated_by, validated_at
```

**Формы контроля** — пять (или шесть) фактов (§13).

**Педагогический бонус** — отдельный факт (§14).

Все поля append-only. Изменения — только корректирующими документами.

## 16. Педагогическая проекция карточки события — НОВОЕ В v0.3

Карточка события — одна сущность с двумя проекциями (студент и
педагог). Педагогическая проекция предоставляет педагогу полный
контроль над событием в зависимости от состояния.

### 16.1. Четыре зоны

```
┌──────────────────────────────────────────────────┐
│  Шапка: дисциплина, дата, состояние, кнопки      │
├──────────────────────────────────────────────────┤
│ [Обзор] [Материалы] [Участники] [Формы контроля] │
├──────────────────────────────────────────────────┤
│                                                  │
│  Содержимое выбранной вкладки                    │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 16.2. Вкладка «Обзор»

- Метаданные события: дисциплина, вид, дата/время, формат, место,
  группы, ссылка на пакет.
- Состояние события (текущее из шести).
- Кнопки управления состоянием (контекстные по состоянию):
  - В `запланировано` — «Заявить о переносе».
  - В `регистрация_закрыта` — «Начать занятие» (требует
    подтверждения присутствия).
  - В `идёт` — «Закончить занятие».

### 16.3. Вкладка «Материалы»

То же, что у студента (через `visibility_gate`), плюс права педагога:
- Разблокировка `teacher_unlocked` материалов.
- Работа с overlay пакета (на состояниях до `завершено`).

### 16.4. Вкладка «Участники»

- Список группы для события.
- Кто зарегистрировался, кто нет, статусы регистраций
  (pending/confirmed/rejected).
- Кнопки валидации/отклонения регистраций (в окне валидации).
- На `идёт` — индикатор «в работе сейчас» по каждому студенту.

### 16.5. Вкладка «Формы контроля»

- Банк форм события из пакета.
- На всех состояниях видна, кнопки запуска контекстные (активны
  только в `идёт`).
- Это позволяет педагогу готовиться к занятию заранее.
- В `идёт` превращается в **runtime-пульт** (§17).

### 16.6. Доступность кнопок управления по состояниям

| Состояние | Обзор | Материалы | Участники | Формы контроля |
|---|---|---|---|---|
| `запланировано` | ✓ (+ «Заявить о переносе») | ✓ (overlay) | ✓ (просмотр) | ✓ (просмотр банка) |
| `регистрация` | ✓ | ✓ | ✓ (pending-регистрации) | ✓ (просмотр) |
| `регистрация_закрыта` | ✓ (+ «Начать занятие») | ✓ | ✓ | ✓ (просмотр) |
| `идёт` | ✓ (+ «Закончить») | ✓ (+ разблокировка) | ✓ (с индикаторами) | ✓ (runtime-пульт) |
| `завершено` | ✓ (read-only) | ✓ (read-only) | ✓ (итог + валидация в окне) | ✓ (статусы пяти фактов) |
| `перенесено` | ✓ (статус) | ✓ (read-only) | ✓ (зафиксированный состав) | — |

### 16.7. Дефолтная вкладка по точке входа

| Точка входа | Дефолтная вкладка |
|---|---|
| Дашборд `event` | Обзор |
| Расписание | Обзор |
| Карточка куратора → событие | Участники |
| Дашборд `teacher_event_debt` | Формы контроля (с подсвеченной проблемой) |

## 17. Runtime-пульт педагога в `идёт` — НОВОЕ В v0.3

Поверхность вкладки «Формы контроля» в состоянии `идёт`. Три зоны —
разные ракурсы одних и тех же данных (model — пять фактов).

### 17.1. Зона 1 — банк форм события

Список форм из пакета, в порядке методического сценария.

По каждой форме:

- Название, тип (тест / эссе / устный / контрольная / лабораторная).
- Класс (обязательная / вариативная / бонусная).
- Time_model (inline / extended) с обозначением.
- Для inline — `duration_minutes`.
- Доступные cohort-операции.
- Запас времени (для inline): «успеете +12 мин» / «не уложитесь −5
  мин» (§11.6).
- Статус: «не запущена / запущена для всех / запущена частично /
  завершена».
- Кнопки запуска (контекстно по классу формы).

**Отдельная секция «Подготовленное от студентов»** — pre_event-бонусы,
которые студенты взяли (pickup). По каждому видно: кто взял, готов
ли. Педагог запускает их в нужный момент.

### 17.2. Зона 2 — поток текущей работы

Динамический список того, что прямо сейчас запущено и идёт.

Для каждой запущенной формы:

```
[Контрольная по теме 3]  обязательная, всем, inline 30 мин
запущена 10:32 · истечёт 11:02 (осталось 12 мин)

Прогресс:
  ✓ Сдали — 8 из 18
  ⏳ В работе — 7 из 18
  ◯ Не приступили — 3 из 18
```

Для индивидуальных вызовов (`cohort: single`) запись компактнее.

Для extended-форм — нет таймера, есть указание дедлайна для каждого
участника.

### 17.3. Зона 3 — состав участников

Список зарегистрированных студентов. По каждому — текущее состояние
по событию: какие формы назначены/запущены, на каком этапе пяти
фактов. Для точечного воздействия педагога.

### 17.4. Polling — 60 секунд авто + жест

Автообновление пульта раз в 60 секунд. Плюс жест «обновить сейчас».
Никакого суперонлайна. Педагог не диспетчер реального времени.

### 17.5. Запуск форм — действия

- **Обязательная форма:** одна кнопка «Запустить всем».
- **Вариативная форма (assignment_group):** две кнопки —
  - «Назначить всем одинаково» (массовое применение группы).
  - «Вызвать студента» (индивидуальный launch).
- **Бонус (pre_event, взят):** кнопка «Запустить (предоставить
  слово)».
- **Бонус (post_event):** не отображается в runtime-пульте (он
  запускается студентом после `завершено`).

### 17.6. Защита от переработки

Перед запуском inline-формы пульт проверяет запас времени (§11.6).
Если отрицательный — кнопка требует осознанного подтверждения.
Журналируется `launched_with_overrun`.

### 17.7. После `завершено`

Пульт уходит в read-only режим. Видны статусы пяти фактов по каждой
запущенной форме. Завершение inline-форм автоматическое (§11.6).
Extended продолжают идти.

## 18. Студенческая проекция карточки события

Тот же набор вкладок (Обзор / Материалы / Участники / Формы контроля),
но с правами и составом студента:

- **Обзор:** свои данные о событии. На состояниях `регистрация`,
  `регистрация_закрыта` доступна кнопка «Отметиться» (через
  presence verification). На `идёт` — отметка о статусе своей
  регистрации.
- **Материалы:** по visibility_gate, как описано в §10.
- **Участники:** простой список одногруппников (без модерационных
  действий).
- **Формы контроля:** свои активные формы (запущенные педагогом, или
  доступные для самостоятельного запуска post_event-бонусы) и
  принципиально допустимые формы события (для информирования).

Дополнительно: подача апелляции на оценку — функция внутри карточки
результата формы контроля (не отдельная вкладка карточки события).
