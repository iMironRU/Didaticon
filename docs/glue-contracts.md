# Контракты клея (фасад слоя активности): приём прохождения, граница события, проводка свидетельства

**Статус:** Рабочая гипотеза — открыта для уточнения.
**Версия:** 0.1.
**Слой:** доставка/рантайм — между плеером (scorm-again) и Univerkon.
**Охват сейчас:** только SCORM (cmi5 отложен).
**Принадлежность:** iMironRU/edu-framework. Реализационная концепция.

---

## 0. Как читать

Клей — фасад слоя активности (того, что звали LRS) между плеером SCORM-контента и ядром Univerkon. Он принимает прохождение, держит живое черновое за фасадом, на границе учебного события лепит свидетельство и проводит его в Univerkon.

Ключевое свойство: **Univerkon видит только свидетельства — ни CMI, ни xAPI.** Хранилище за фасадом — приватная сменяемая деталь (на SCORM-only старте — тонкий CMI-снимок; cmi5 придёт — подменишь на SQL LRS, не трогая контракты).

## 1. Несущие принципы

1. **Univerkon видит только свидетельства.** Форма хранения активности за фасадом наружу не протекает.
2. **Клей вне горячего пути.** Буферизован с обеих сторон: сверху — outbox PWA, снизу — outbox-в-Univerkon. Недоступность ядра или самого клея учащемуся невидима.
3. **Попытка = новая запись.** Клей не правит факты — каждый исход это новая запись.
4. **Закрытость вычисляет Univerkon, не клей.** Клей ставит валентность записи (положительная/отрицательная); закрытость (§5) и пересдачу (§11) считает ядро.
5. **Граница ≠ коммит.** Свидетельство рождается на терминальном исходе учебного события, а не на каждом CMI-коммите.

## 2. Контракт к PWA

Два метода:

### `commit`
Положить порцию CMI.
- **Вход:** `{ event_id, attempt_id, sequence, cmi, status?, credential }`, где `event_id` — учебное событие из лонч-контекста; `attempt_id` — присвоен при запуске; `sequence` — монотонный счётчик коммита в попытке.
- **Аутентификация:** токен PWA (OIDC) или серт станции (mTLS, локальная зона) — `credential`.
- **Идемпотентность:** ключ дедупа `(event_id, attempt_id, sequence)`. Повтор при ретрае — без эффекта.
- **suspend_data:** last-write-wins в пределах `(event_id, attempt_id)`.

### `resume`
Отдать последнее состояние попытки для возобновления.
- **Вход:** `{ event_id, attempt_id, credential }`.
- **Выход:** последнее синхронизированное CMI-состояние попытки или пусто.
- **Ограничение:** кросс-девайс resume работает только для **синхронизированного** состояния. Начал офлайн на устройстве A, не синкнул, пересел на B → на B пусто. Несинхронизированный буфер живёт в IndexedDB устройства A.

## 3. Контракт к Univerkon

Один метод — JSON-RPC + JWT (паттерн КандиМил), сервисный токен клея.

### `deposit_svidetelstvo`
Положить свидетельство (черновое) на учебное событие.
- **Вход:** `{ event_id, student_id, attempt_id, valence, status, score?, occurred_at }`
  - `valence` — `positive` | `negative`;
  - `status` — исходный SCORM-статус (для аудита);
  - `score` — сырой балл (свёртку по формуле §7.2 делает мост Univerkon, не клей);
  - `occurred_at` — момент завершения у конкретного студента.
- **Идемпотентность:** одно свидетельство на `(event_id, attempt_id)`. Повтор — без дублей.
- **Univerkon далее:** создаёт запись как черновое; вычисляет закрытость (§5); при `negative` запускает логику пересдачи/задолженности (§11); валидатор затверждает черновое→факт (§15.8). Это уже ядро, не клей.

## 4. Правило границы события

Клей фиксирует свидетельство, когда исход **терминальный и определённый** для формы контроля узла. Семантика формы контроля приходит в лонч-контексте (PWA знает узел и его форму); дефолт — `passed`.

| Тип узла | Терминальный исход | Валентность |
|---|---|---|
| Контентный («просто пройти», закрытие = completion) | `completed` | положительная (закрывает) |
| Оценочный (закрытие = pass) | `passed` | положительная (закрывает) |
| Оценочный | `failed` | **отрицательная** (зафиксированная попытка, НЕ закрывает → пересдача §5/§11) |
| Любой | `incomplete` / suspended | свидетельство **не** лепится — это живое черновое (resume) |

Существенное (правка по обсуждению): клей лепит запись и на **отрицательный** исход. Failed — это не «ничего», а зафиксированная попытка, которая по §5/§11 рождает пересдачу. «Закрывает» и «рождает запись» — разные вещи: запись лепится на любой терминальный исход, а закрывает только положительная.

## 5. Попытка vs resume

Диктует форма контроля узла:

- **Экзамен / аттестация** — каждый запуск это новая попытка, resume нет. Провал (отрицательная запись) → Univerkon порождает пересдачу = новую попытку (вторая — комиссионная, §11.5).
- **Контентный узел** — resume: повторный запуск продолжает ту же попытку из suspend_data.

`attempt_id` присваивается при запуске (PWA/лонч): новая попытка → новый `attempt_id`; resume → прежний. Счётчик попыток у Univerkon = число записей по обязательству (§5).

## 6. Хранилище — за фасадом

Сменяемая внутренняя деталь. На SCORM-only — тонкий CMI-снимок на `(event_id, attempt_id)` без трансляции. Контракт хранения держать xAPI-формы про запас: придёт cmi5 — подменишь на SQL LRS, контракты §2–§3 не меняются.

## 7. Устойчивость

- Буферы с обеих сторон (§1.2): лёг Univerkon — свидетельство ждёт в нижней очереди; лёг клей — коммиты ждут в outbox PWA.
- Клей рестартуемый, очереди durable.
- На кампусе клей разворачивается edge-узлом (тонкий store/SQLite) для локального экзамена и сводит свидетельства вверх по возвращении канала.
