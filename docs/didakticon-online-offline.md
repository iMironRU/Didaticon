## Как читать этот документ

Документ описывает границу между online и offline операциями
Дидактикона как PWA. Для каждой функции зафиксировано, в каком режиме
она работает, что показывает пользователю при отсутствии сети, как
ведёт себя при разрыве по ходу выполнения.

Документ — спецификация для PWA-команды. Архитектурные решения
зафиксированы.

---

## 1. Назначение

Дидактикон — PWA, mobile-first. Сеть мобильного устройства
нестабильна: тоннели, лифты, аудитории-«бункеры», метро, плохой
оператор в зоне университета. Студент или педагог не должны терять
данные и работоспособность приложения из-за временного разрыва.

Документ решает три задачи:

1. Дать понятную классификацию операций по требованию к сети.
2. Описать поведение каждого жанра функции при потере связи.
3. Зафиксировать стратегии кеширования, прелоадинга и буферизации.

---

## 2. Несущие принципы

1. **Источник истины — на сервере.** Универкон, Тестикон, шина —
   единственные хранители фактов. Клиент не может породить факт
   локально, минуя сервер.
2. **Клиент компенсирует разрывы, не подменяет сервер.** Все
   локальные кеши, черновики, буферы — это **подстраховка** на время
   отсутствия сети, не альтернативный источник.
3. **При потере данных приоритет — НЕ потерять работу пользователя.**
   Если студент 20 минут писал ответ и сеть отвалилась — приложение
   должно сохранить и доставить, не показывать «начни заново».
4. **При невозможности продолжить — честно сообщить.** Никаких
   тихих сбоев, никакой работы «как будто всё ок» при недоступности
   сети для критичных операций.
5. **Кеш всегда помечен датой свежести.** Пользователь видит, что
   данные не «прямо сейчас», а «с такого-то времени».
6. **e-Student — особая стратегия:** офлайн-валидация по подписи,
   проактивное обновление при сети.

## 3. Три класса операций

### 3.1. ONLINE-ONLY

Требует сеть, без неё функция недоступна или ограничена. Все
операции, которые:
- Создают или меняют факт в Универконе/Тестиконе.
- Требуют свежей серверной валидации (presence, аутентификация).
- Открывают внешние приложения.

### 3.2. OFFLINE WITH CACHE

Работает с закешированными данными, помеченными временем последнего
обновления. Пользователь может смотреть, но не действовать (на
изменение факта).

### 3.3. OFFLINE FIRST

Работает без сети по своей природе. Не зависит от свежести. Кеш —
основной режим, не fallback.

## 4. Карта операций по классам

### 4.1. ONLINE-ONLY

| Операция | Где | Что при потере сети |
|---|---|---|
| Логин (OIDC flow) | Универкон | «Нет сети. Войти невозможно» |
| Refresh токена | Универкон | Графдowный logout с пометкой «потеря сессии при возвращении» |
| Сборка свежего дашборда (`feed.get`) | Дидактикон-backend | Показывается последний кеш (см. §4.2) |
| Регистрация на занятии (presence verification) | Дидактикон-backend | «Нет сети — обратитесь к преподавателю». Преподаватель отмечает вручную из своего пульта |
| Запуск формы контроля педагогом | Дидактикон-backend | Кнопка неактивна с пояснением. Без сети запустить нельзя |
| Сдача попытки студентом | Тестикон через прокси | См. §5 (специальная обработка разрыва по ходу) |
| Оценивание (фиксация записи) | Тестикон через прокси | Локальный черновик сохраняется. См. §6 |
| Раздача педагогических бонусов | Универкон | Не сохраняется без сети, явная пометка |
| «Галочка я всё» (завершение работы с модулем) | Универкон | Не отправляется без сети |
| Подача апелляции / переопределения | Дидактикон-backend | Не отправляется без сети, форма сохраняется как черновик локально |
| Открытие внешнего приложения (`external_action`) | Deep-link | Deep-link срабатывает без сети (если приложение установлено), fallback URL — требует сеть |
| Выпуск свежего e-Student | Универкон | См. §7 |
| Прокси к Тестикону (любые запросы) | Тестикон | Возвращает кеш с пометкой устаревания |

### 4.2. OFFLINE WITH CACHE

Все эти данные кешируются на стороне PWA с пометкой времени последнего
обновления. При отсутствии сети PWA показывает кеш + банер «Нет сети.
Данные обновлены: <время>».

| Операция | Стратегия кеша | TTL свежести |
|---|---|---|
| Просмотр дашборда | last `feed.get` ответ | 60 секунд (затем пометка «устарело») |
| Карточка события (метаданные) | при первом открытии + предзагрузка для расписания ±N дней | до изменения от сервера |
| Расписание (горизонт 2 недели) | предзагрузка при заходе на экран расписания | сутки |
| БРС / траектория / зачётка | при первом открытии + обновление по жесту | до закрытия следующего модуля |
| Карточка результата (оценённая работа) | при первом открытии | до закрытия следующего модуля |
| Очередь работ на проверке (у педагога) | last `submissions.pending` ответ | 5 минут |

### 4.3. OFFLINE FIRST

Работают без сети.

| Операция | Хранилище | Примечание |
|---|---|---|
| UI приложения (shell, навигация, экраны) | service worker cache | Стандартный PWA app shell |
| e-Student QR (визуальное предъявление) | localStorage / IndexedDB | См. §7 |
| Чтение материалов пакета (если предзагружены) | service worker cache + IndexedDB | См. §8 |
| Локальные черновики оценок педагога | IndexedDB | См. §6 |
| Локальные черновики ответов студента | IndexedDB | См. §5 |
| Личные настройки (тема, язык, флаги) | localStorage | — |
| Локальная маршрутизация между экранами | в памяти приложения | — |

## 5. Сдача формы контроля при потере сети по ходу выполнения

Студент пишет тест, эссе, лабораторную. На середине отвалилась сеть.

### 5.1. Стратегия: продолжать на клиенте, буферизовать

PWA не останавливает попытку. Продолжает таймер локально. Ответы
буферизуются в IndexedDB по мере ввода (автосохранение каждые N
секунд бездействия).

Студент видит небольшой индикатор «Нет связи, ответы сохраняются
локально» в углу экрана. Не модальное окно, не блокирующее
поведение — индикатор статуса.

### 5.2. При появлении сети

PWA в фоне отправляет накопленные ответы в Тестикон. Если попытка
ещё не сдана — синхронизация состояния. Если истёк дедлайн (см. §5.3)
— отправка как `expired_during_attempt` с буфером данных.

### 5.3. Если дедлайн истёк во время офлайна

Попытка завершается на клиенте по локальному таймеру. PWA сохраняет
последнее состояние ответов и помечает как **`expired_during_attempt`
с локальной буферизацией**.

При появлении сети:
- Отправка в Тестикон с пометкой `was_offline: true`,
  `offline_duration_seconds: N`.
- Тестикон фиксирует попытку как `submission` со статусом
  `expired_during_attempt`, но с **доставленным контентом**.
- Педагог при проверке видит работу + пометку «частично выполнено в
  офлайне, дедлайн истёк во время разрыва связи».

Решение по оценке — за педагогом. Это **не автозачёт** — это
сохранение работы для рассмотрения.

### 5.4. Если событие закончилось во время офлайна (для inline-форм)

Inline-формы принудительно завершаются с концом события (см. §11.6
блока III). Если студент в этот момент офлайн:

- Локальный таймер inline-формы знает `event.planned_end_at` (он
  передан при `launch`).
- В момент `planned_end_at` локально срабатывает принудительное
  завершение попытки на клиенте.
- Ответы буферизуются как `expired_at_event_end`.
- При появлении сети — отправка в Тестикон с пометкой.

Студент не «доходит тест в коридоре» физически — он мог продолжать
писать локально, но попытка на сервере будет помечена как
истёкшая. Принцип защиты от опоздания на следующую пару не
нарушается (записи валидируются педагогом).

### 5.5. Журнализация офлайна

В факте `submission` появляются дополнительные поля:

```
submission:
  ...
  was_offline: bool
  offline_duration_seconds: int
  offline_at: ISO8601 (момент потери сети)
  delivered_at: ISO8601 (момент доставки на сервер)
```

Это даёт аналитику ОО и педагогам — насколько часто случаются
разрывы при попытках, какие сегменты сети проблемны.

## 6. Локальные черновики оценок педагога

Педагог в разделе «Работы на проверке» оценивает работу. Может
поставить балл, написать комментарий, но **не нажал «Принять
оценку»** — оценка ещё не отправлена.

### 6.1. Стратегия

Черновики хранятся **локально в IndexedDB** до явного «Принять».
Никакого автосохранения на сервер. Принцип «факт неизменяем» —
черновик не должен становиться фактом без явного действия педагога.

Локальные черновики — пер-устройство. Открыл работу на планшете,
поставил оценку, не нажал «Принять» → закрыл планшет → открыл
компьютер → черновика на компьютере **нет**. Это сознательное
ограничение, не баг: смешение черновиков с разных устройств опаснее,
чем требование «принимать оценку с одного устройства за сеанс».

### 6.2. При потере сети в момент принятия

Педагог жмёт «Принять оценку» в момент офлайна. PWA:

1. Сохраняет оценку как черновик в IndexedDB с флагом
   `pending_send: true`.
2. Показывает индикатор «Сохранено локально, ожидает отправки».
3. При появлении сети — отправка в Тестикон в фоне.
4. Если Тестикон вернул успех — черновик удаляется, в UI «Принято».
5. Если Тестикон вернул ошибку (например, работа уже оценена
   другим педагогом) — конфликт показывается пользователю.

### 6.3. При закрытии приложения с непринятыми черновиками

При следующем открытии раздела «Работы на проверке» PWA показывает
**отдельную секцию «Не отправленные оценки»** с возможностью
просмотреть и подтвердить каждую отдельно.

## 7. e-Student — стратегия обновления и валидации

### 7.1. Свежесть

JWS-токен e-Student живёт **сутки** (24 часа от выпуска). Универкон
выпускает новый по запросу `identity.e_student.issue`.

### 7.2. Проактивное обновление

При каждом запуске приложения **с сетью** PWA в фоне проверяет срок
текущего токена для каждого активного обучающегося контекста. Если
до истечения меньше **12 часов** — запрос на обновление в Универкон.

Это происходит **прозрачно**, без визуальной активности. Студент в
течение дня всегда имеет свежий или почти свежий e-Student.

### 7.3. Поведение при отсутствии сети

- Если токен **актуален** (не истёк) — карточка показывается
  нормально, QR валиден.
- Если токен **истёк** во время офлайна — карточка показывается с
  пометкой «срок истёк, обновите при появлении сети». Охранник или
  валидатор может принять решение по своему усмотрению (узнал
  студента визуально и т. п.).

### 7.4. Офлайн-валидация QR проверяющим

Сканирующая сторона (приложение охранника, киоск библиотеки)
валидирует подпись по публичному ключу Универкона.

Публичный ключ должен быть доступен валидатору:
- В режиме онлайн — через `/jwks` или
  `/.well-known/e-student-keys`.
- В режиме офлайн — приложение валидатора **кеширует** ключ при
  каждом подключении.

Это валидаторская задача, не Дидактиконовская — Дидактикон только
выдаёт корректно подписанный JWS.

## 8. Прелоадинг материалов пакета

### 8.1. Стратегия — Lazy + Service Worker cache

PWA не предзагружает все материалы пакета автоматически. Слишком
дорого по трафику и по хранилищу мобильного устройства.

При первом открытии материала студент скачивает его из сети.
Service worker кеширует. При следующем открытии (включая
офлайн) — материал доступен.

### 8.2. Прелоадинг карточек событий

При входе на экран расписания PWA загружает **метаданные** всех
событий на горизонте 2 недели и **первый уровень материалов пакета**
(только структура, не файлы). Это позволяет открыть карточку события
офлайн и увидеть, что в ней есть (даже если самих материалов ещё
нет в кеше).

### 8.3. Принудительное скачивание (опционально, не MVP)

В будущей итерации — кнопка «Скачать для офлайн» на карточке
события, которая запускает eager prefetch всех материалов пакета.
Полезно при планируемом отсутствии сети (поездка, командировка). В
MVP — не делаем.

### 8.4. Очистка кеша

Service worker кеширует материалы. По мере роста кеша — выселение по
LRU (least recently used). Лимит на устройстве — порядка 50 МБ
(настройка PWA), управляется автоматически.

Материалы завершённых событий (`завершено` более 30 дней назад)
помечаются как кандидаты на выселение в первую очередь.

## 9. Поведение приложения при разных состояниях сети

### 9.1. Online

Все функции доступны. Никаких баннеров.

### 9.2. Offline (полная потеря сети)

- В шапке приложения — банер «Нет сети. Часть функций недоступна».
- ONLINE-ONLY кнопки — серые с подсказкой при тапе.
- OFFLINE WITH CACHE экраны — показываются с пометкой свежести.
- OFFLINE FIRST функции — работают как обычно.

### 9.3. Slow / unreliable network

- Если запрос идёт дольше N секунд (например, 10) — показывается
  индикатор «Медленная сеть».
- Запрос продолжается в фоне. Если успел до timeout — ответ. Если
  нет — fallback на кеш с пометкой.

### 9.4. Восстановление сети

- Service worker автоматически выполняет отложенные операции
  (sync events, см. §10).
- В UI — недолгий банер «Сеть восстановлена».
- Перезапрос свежих данных (дашборд, активные попытки).

## 10. Background Sync (Service Worker)

Для отложенной отправки данных используется Background Sync API.

### 10.1. Очередь отложенных операций

Все ONLINE-ONLY операции, инициированные пользователем без сети,
ставятся в **очередь отложенных** в IndexedDB:

```
deferred_operation:
  id: uuid
  method: string (JSON-RPC method name)
  params: object
  created_at: ISO8601
  status: pending | sending | sent | failed
  retry_count: int
  last_error?: string
```

### 10.2. Приоритеты

Не все отложенные операции равнозначны:

- **Высокий:** submission попытки (риск потерять студенческую
  работу).
- **Средний:** evaluation, pedagogical_bonus (можно подождать).
- **Низкий:** аналитические события (которые не блокируют UX).

Background Sync обрабатывает в порядке приоритета.

### 10.3. Конфликты при отложенной отправке

Если за время офлайна состояние на сервере изменилось (например,
работа уже оценена другим педагогом — для senior_grader, открыта
апелляция и т. п.):

- PWA получает ответ с ошибкой `conflict`.
- Операция остаётся в очереди со статусом `failed` + пометка
  конфликта.
- В UI пользователю показывается уведомление с возможностью
  посмотреть конфликт и решить (вручную).

### 10.4. Лимит хранения

Очередь отложенных операций — до 100 элементов. При переполнении —
самые старые операции **не удаляются автоматически**, требуется
явное действие пользователя «Очистить очередь» с подтверждением.

## 11. Что в этой версии не описано

1. **Принудительное скачивание материалов** «для офлайн» — будущая
   итерация.
2. **Глобальная индикация состояния** очереди отложенных операций —
   деталь UX, согласовывается с PWA-командой.
3. **Push-уведомления** через Service Worker — не используются в
   MVP (push не нужен по принципу 7 блока II).
4. **Multi-device synchronization** локальных черновиков —
   сознательно не реализуется (см. §6.1).

## 12. Что синхронизировать с командами

### 12.1. С Универконом

1. Сроки `e-Student` (24 часа), окно проактивного обновления (12
   часов).
2. Поддержка `was_offline: true` в `submission`-фактах.
3. Возврат `conflict`-ошибок при коллизиях отложенной отправки.

### 12.2. С Тестиконом

1. Принятие `submission` с буферизованным контентом и пометкой
   `was_offline`.
2. Семантика `expired_during_attempt` — попытка с истёкшим дедлайном,
   но с доставленным контентом (отличается от `expired_no_attempt`).

### 12.3. С PWA-командой

1. Service Worker — caching strategies (network first / cache first
   по типам ресурсов).
2. IndexedDB — схемы хранения (попытки, черновики, отложенные).
3. Background Sync API — регистрация sync events.
4. App Shell для базовой навигации без сети.
