# Пакет занятия для ЭИОС Univerkon — концепция

Рабочий документ. Фиксирует архитектурные решения, принятые в ходе обсуждения. Правится итеративно.

**Статус:** черновик v0.1, MVP-фокус
**Контекст:** Univerkon / Дидатикон (ЭИОС PWA), edu-framework

---

## 1. Что такое пакет

Пакет — **манифест занятия**, навигационно-контекстная оболочка для студента, а не контейнер учебных материалов.

### Природа пакета

- **Лёгкий и эмбедет.** Никаких бинарных файлов внутри, только описание и ссылки на материалы. Размер — десятки–сотни КБ.
- **Стейтлесс.** Пакет ничего не помнит между запусками. Любое состояние (прогресс студента, факт прохождения gate, оценки) живёт исключительно в LMS.
- **Атомарен.** Один пакет = одна единица учебного процесса: одно занятие, один экзамен, один день практики и т.д. Никаких «count».
- **Переиспользуем.** Один и тот же пакет применяется для всех групп, изучающих эту тему в этой РПД, в любом семестре, пока РПД актуальна.

### Привязка пакета

Пакет привязан к:
- образовательной программе (ОП)
- учебному плану
- РПД/РПП конкретной дисциплины
- теме тематического плана
- виду занятия

Пакет **не привязан** к:
- группе студентов
- дате проведения
- конкретному преподавателю
- семестру
- формату проведения (online/offline/mixed)

Всё непривязанное приходит как **runtime-контекст от LMS** при запуске пакета.

## 2. Три источника данных

| Слой | Источник | Заполняется когда | Примеры полей |
|---|---|---|---|
| **Base** | ИС вуза (Univerkon) | При генерации пакета | ОП, РПД, тема, компетенции, базовая литература |
| **Overlay** | Педагог | После генерации, при обогащении | Цели занятия, материалы, задания, подсказки, gate-тексты |
| **Runtime** | LMS при запуске | На лету | Группа, преподаватель, дата, формат, идентичность студента |

**Принцип:** base и overlay живут в самом пакете как два слоя, при рендеринге сливаются. Регенерация пакета обновляет base, overlay сохраняется.

## 3. Стратегический разворот MVP

**Главная задача MVP — не «дать педагогу инструмент», а наполнить готовую ЭИОС пакетами автоматически из ИС.**

### Следствия

- Пакет валиден и полезен **даже когда от педагога ничего нет**.
- Builder = генератор данных + формирователь TODO для педагога.
- Конструктор для педагога — задача v2.
- Версия пакета — индикатор зрелости:
  - **0.x** — авто-скелет из ИС
  - **1.x** — авторизован педагогом
  - **1.x+** — итерации совершенствования

### Что в ИС уже есть на момент Х

- Структура учебного процесса (ОП → УП → РПД → тематический план → темы) — D06
- Расписание — D04
- Кадры — D02
- Контингент — D03
- Литература РПД
- Компетенции дисциплины

### Что обычно дописывает педагог

- Цели именно этого занятия (vs дисциплины в целом)
- Слайды, видео, методички (ссылки)
- Задания на практику
- Дополнительная литература
- Подготовка к занятию
- Канал связи курса
- Gate'ы
- Самопроверочные тесты

## 4. Структура lesson.json

### Общие разделы (для всех видов)

| Раздел | Содержимое |
|---|---|
| `provenance` | Паспорт пакета: package (id, version, revision), educational_program, curriculum, syllabus, topic |
| `event` | kind + title |
| `duration` | unit: lesson \| pair \| academic_day. Всегда **одна** единица. |
| `didactic` | objectives, competencies, prerequisites, expected_outcomes |
| `delivery` | supported_formats + format_notes для каждого формата |
| `gates.entry` | Входные шлюзы |
| `gates.exit` | Выходные шлюзы |
| `materials` | main, additional, bibliography (with from_curriculum + extra) |
| `communication` | Каналы связи курса |
| `completion` | Что считать прохождением |

### Provenance — паспорт пакета

```yaml
provenance:
  package:
    id: "pkg-bd-3nf"             # стабильный
    version: "1.4.0"             # semver редакции методики
    revision: "git-sha-7f3a2c1"  # конкретная сборка
    created_at: ...
    created_by: ...
    schema_version: "1.0"
  educational_program:
    code, title, level, standard, ref
  curriculum:
    ref                          # без года/семестра
  syllabus:
    kind: rpd|rpp|rpo_dpo
    discipline_code, discipline_title, ref
  topic:
    code, title, ref
```

### Duration

```yaml
duration:
  unit: lesson | pair | academic_day
```

- `lesson` — академический час (45 мин)
- `pair` — пара (90 мин с переменой внутри)
- `academic_day` — учебный день (для практики, экзамена и т.п.)

Единица одна. Если занятие двухчасовое — это `pair`, а не `{lesson, 2}`. Если день — `academic_day`.

### Delivery

```yaml
delivery:
  supported_formats: [offline, online, mixed]
  format_notes:
    offline:
      bring_with_you: [...]
      dress_code: ...
    online:
      requirements: [...]
      backup_plan: ...
    mixed:
      note: ...
```

Формат проведения **не зашит в пакете** — определяется LMS при запуске. Пакет лишь декларирует, какие форматы поддерживает.

## 5. Gate — первоклассная сущность

Gate — шлюз, блокирующий проход по занятию, пока условие не выполнено.

### Виды gate

| Вид | Семантика | Примеры |
|---|---|---|
| `entry-test` | Проверка готовности через тест | Допуск к лабораторной по теории |
| `consent` | Юридически значимое волеизъявление | Согласие на видеозапись, на обработку ПД |
| `briefing` | Подтверждение ознакомления | Инструктаж по ТБ, регламент дисциплины |
| `readiness` | Самодекларация готовности | «Прочитал главу 5, готов к семинару» |
| `form` | Заполнение данных | Декларация наличия СИЗ, выбор варианта |
| `attendance` | Подтверждение присутствия | QR-код / геолокация / код из чата |

### Размещение в занятии

- **Entry** — перед началом, без прохождения нельзя начать
- **Exit** — перед завершением, без прохождения нельзя завершить
- **Interstitial** — между смысловыми частями (отложено, не MVP)

### Принципы

- Пакет **стейтлесс**: факт прохождения gate целиком в LMS.
- При запуске пакет спрашивает LMS «эти gate'ы уже пройдены этим студентом?» через launch context.
- Тексты gate'ов — преимущественно из централизованной библиотеки (вуз/кафедра), ad-hoc — исключение.
- **Версионирование текстов обязательно.** Факт прохождения привязан к версии текста.

### Юридический контекст

- Инструктажи по ТБ часто требуют **бумажной подписи в журнале**. Электронная отметка её не заменяет, но может быть предварительным шагом.
- Согласия и инструктажи — пересекаются с D? «Адресные коммуникации» (TSP-таймстампинг, аудит).

## 6. Виды занятий

### Горизонт (определяется D04 Univerkon, мы предусматриваем)

**ВО:** lecture, seminar, practice, lab, consultation, colloquium, control_work, exam, credit, placement, coursework_defense

**СПО добавляет:** lesson_combined, lesson_practice, lesson_lab, lesson_test, course_project_defense, qualification_exam, placement_lesson

**Прочие:** test_session, project_consultation, excursion, olympiad/conference, independent_work

**Принцип:** перечень видов закрытый, определяется на уровне ИС, в пакете — ссылка из этого справочника. Каждый вид имеет свой профиль: набор обязательных/опциональных разделов, типичные gate'ы, типичные активности.

### MVP — два вида: lecture и practice

## 7. Lecture (лекция)

### Назначение
Студент получает новый материал, осмысливает, опционально проверяет понимание.

### Длительность
`pair` (типично) или `lesson` (СПО).

### Специфичные разделы

- **`lecture_plan`** — план изложения (навигация по содержанию)
- **`key_concepts`** — глоссарий ключевых понятий с привязкой к глоссарию дисциплины
- **`further_reading`** — что почитать после, для углубления

### Активности (read+listen-only)

- `listen_lecture` / `watch_recording` — слушать или смотреть запись
- `read_materials` — прочитать слайды/конспект
- `self_check_test` — пройти тест самопроверки (опц.)

### Что фиксируется как факт прохождения

`completion.required` для лекции — `[opened]` или `[opened, materials_viewed]`. Лекция — не зачёт.

### Отложено для лекции

- **Интерактивный конспект** (allow_notes) — не MVP.

## 8. Practice (практическое занятие)

> **Важно:** practice ≠ практика. Практическое занятие — урок/пара решения задач. Производственная/учебная практика — отдельный вид (`placement`), один пакет = один день.

### Назначение
Студент решает задачи, разбирает кейсы, отрабатывает навык.

### Длительность
`pair` (типично).

### Специфичные разделы

- **`tasks`** — задания (ключевой раздел). Структура:
  - id, title, kind (individual/group/choice), difficulty, points, time_estimate
  - description_md (условие)
  - hints (с правилами раскрытия по времени)
  - solution_approach_ref (методика; visible_when: immediate/after_submission/never)
  - submission (kind: text/file/both, формат, размер, word_count)
  - assessment (kind: auto/manual/hybrid, rubric_ref, auto_check)

- **`tasks_distribution`** — политика распределения заданий между студентами:
  - `all` — все задания всем
  - `one_per_student` — LMS назначает одно
  - `choice_n_of_m` — студент выбирает N из M
  - `by_variant` — варианты
  - **Реальное распределение — runtime в LMS, в пакете только политика.**

- **`solution_approach`** — методика/алгоритм решения (с правилом visible_when)

- **`assessment_criteria`** — общие критерии за занятие (рубрика + порог + шкала)

- **`examples`** — разобранные примеры

### Активности

- `solve_task`, `submit_solution`
- Опц.: `discuss_solution`, `review_peer`, `self_assess`, `consult_teacher`

### Что фиксируется

- Сдача обязательных задач
- Оценки (per task + аггрегат)

`completion.required` для практики — `[opened, tasks_submitted, ...]`.

## 9. Тесты и защита тестирования

### Принцип

**Тесты не лежат в пакете.** В пакете — **спецификация выборки** из банка.

```yaml
test_ref:
  bank_url: "univerkon://test-banks/bd-norm-3nf"
  selection:
    count: 10
    difficulty: "mixed"
    weighted: {easy: 3, medium: 5, hard: 2}
    topics: [...]
    exclude_recent_days: 30
  mode: self-check | graded | exam
  time_limit_minutes: 15
  attempts: 3
  passing_scaled: 0.7
  shuffle_questions: true
  shuffle_options: true
  show_results: immediate | after_finish | after_deadline | never
```

### Угрозы и слои защиты

| Слой | Стоимость | Что закрывает |
|---|---|---|
| 1. Серверная выдача вопросов | Дёшево, must have | 80% наивных читов |
| 2. Привязка к сессии (attempt_token) | Дёшево, must have | Повторные открытия, базовая атрибуция |
| 3. Идентификация (SSO + опц. WebAuthn) | Средне | Подмена личности |
| 4. Контекстные ограничения (окно времени, IP) | Средне | Прохождение вне расписания |
| 5. Клиент-сайд античит (CSS-блокировки) | **Не делаем** | Иллюзия защиты |
| 6. Прокторинг (камера, очно) | Дорого | Серьёзные контроли |

### Декларативная политика в пакете

```yaml
integrity:
  mode: self-check | controlled | proctored
  require_sso_identity: true
  time_window: {from, to}
  ip_restriction: any | campus | custom-cidr
  server_side_grading: true
  show_correct_answers: ...
  flag_tab_switch: true
  proctoring: {enabled, provider}
```

Пакет выражает намерение, сервер исполняет.

## 10. Технический подход MVP

### Свой формат, не SCORM

- **Хранение:** git-репозиторий с YAML-файлами пакетов
- **Формат:** YAML с валидацией по JSON Schema
- **Шелл:** отдельное SPA (mobile-first), рендерит данные занятия
- **Связь с LMS:** HTTP API ЭИОС, не cmi-вызовы
- **SCORM/cmi5:** конвертер «когда понадобится», не сейчас

### Структура репо (предварительно)

```
univerkon-packages/
  schema/
    lesson.v1.schema.json
    examples/
  packages/
    bd-2024/
      lec-01-intro.yaml
      lec-03-normalization-3nf.yaml
      prac-01-first-queries.yaml
    web-2024/
      ...
  renderer/                        # шелл, можно вынести в отдельный репо
    index.html
    app.js
    styles.css
```

### Траектория развития

1. Спецификация формата (JSON Schema v1, оба MVP-вида)
2. Один мокап-пакет вручную (например, «Нормализация 3НФ»)
3. Минимальный шелл (mobile-first, статика)
4. Несколько мокап-пакетов (5–10 разных тем)
5. JSON Schema в CI
6. Интеграция с ЭИОС (шелл встраивается, фетчит из git)
7. Параллельно — прототип генератора из Univerkon
8. Массовая генерация на всё расписание

### Резонансы с экосистемой

- **Хроникон** — данные + шаблоны рендеринга
- **fgos2json** — машиночитаемые домены в JSON
- **edu-framework** — открытые спецификации форматов
- **Ярлыкон** — git как source of truth, статика + минимальный API

## 11. Mobile-first инварианты шелла

- single-column layout, max-width ~640px по центру
- viewport + 100dvh, без фиксированных высот
- tap-target ≥44px
- никаких hover-only взаимодействий
- системные шрифты (без подгрузки веб-шрифтов)
- KaTeX для формул заинлайнен
- видео — нативный `<video>`, не embed YouTube
- offline-инвариант: после загрузки пакет работает без сети (кроме материалов по ссылкам)

## 12. Открытые вопросы (для следующих итераций)

### Архитектурные

- [ ] **Активности vs разделы** — дихотомия в схеме. Раздел = описание/данные, активность = действие студента. Зафиксировать формально.
- [ ] **Рубрики и шкалы оценивания** — горизонтальный слой через всю модель, со ссылкой на shared-библиотеку.
- [ ] **Launch context** — формальный контракт между LMS и пакетом. Что обязательно, что опционально, что делает пакет при отсутствии.
- [ ] **Версионность пакета vs bitemporal D06.** Какие версии совместимы, что делает LMS при несовпадении, как помечается устаревший пакет.
- [ ] **Мерж base + overlay** при регенерации пакета. Что обновляется, что сохраняется, как разрешаются конфликты.

### Технические

- [ ] Один git-репо для всех пакетов или по одному на дисциплину (склонились к одному на старте)
- [ ] YAML vs JSON для исходников (склонились к YAML с JSON Schema)
- [ ] Шелл фетчит прямо из git или через API ЭИОС (для мокапов — git, для прода — через ЭИОС)
- [ ] Имя формата в стиле «-кон» или нейтральное

### Расширение MVP

- [ ] Детальный разбор остальных видов занятий (seminar, lab, exam, placement, qualification_exam и т.д.)
- [ ] Интерактивный конспект студента (lecture.allow_notes)
- [ ] Interstitial gate'ы (между смысловыми частями занятия)
- [ ] Подсказки и методика с динамикой раскрытия

### Точки сверки с D-доменами

- [ ] Перечень видов занятий — сверить с D04
- [ ] Поля автозаполнения из РПД — точная сверка с D06
- [ ] Тексты gate'ов и юридическая значимость — пересечение с D? «Адресные коммуникации»
- [ ] Шкалы оценивания — пересечение с D09 / D05

## 13. С чего возобновлять работу

Кандидаты на следующий шаг:

1. **JSON Schema v1** как первый артефакт — формальная спецификация того, что обсудили
2. **Первый мокап-пакет руками** — проверка схемы на жизнеспособность через живой YAML
3. **Активности vs разделы** — закрыть архитектурную дихотомию до Schema
4. **Launch context** — определить контракт с LMS до того, как писать шелл
5. **Имя формата** — мелкий, но влияет на брендинг и ссылки
