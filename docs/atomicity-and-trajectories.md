# Атомарность пакета и траектории обучения

Концептуальный документ. Обосновывает архитектурный выбор «один пакет = одна единица учебного процесса» как фундамент для нелинейного образования.

**Статус:** черновик v0.1
**Контекст:** дополнение к [lesson-package-concept.md](./lesson-package-concept.md)

---

## 1. Тезис

**«Один пакет = одно занятие» — это не ограничение, а фундамент.**

На первый взгляд атомарность звучит как декомпозиция ради декомпозиции: мелкий артефакт, неудобно управлять, фрагментация. На самом деле — наоборот: атомарность открывает то, что монолитные курсы закрывают.

Атомарный пакет — **узел в графе знаний**. Курс-монолит — закрытая упаковка, в которой узлы склеены намертво.

---

## 2. От пакета к графу

### Что такое граф пакетов

Если каждое занятие — самостоятельный переиспользуемый артефакт, то **учебная программа — это путь по графу этих артефактов**, а не их линейная склейка внутри одного контейнера.

Узлы графа — пакеты. Связи — это:
- **prerequisite** — что нужно знать до
- **continuation** — естественное продолжение
- **deepening** — углубление по той же теме
- **alternative** — альтернативная подача того же материала
- **application** — применение в другой области

### Тематический план РПД = один путь в графе

РПД с тематическим планом — это **формально утверждённый обход графа**, продиктованный регулятором (ФГОС) и согласованный методически. Один из возможных. Не единственный.

Другие возможные обходы:

- **стандартная** (по РПД) — для базового потока
- **ускоренная** — для тех, у кого есть знание входных тем
- **углублённая** — с боковыми пакетами уровня «расширенное чтение»
- **индивидуальная** — под академическую разницу при переводе
- **догоняющая** — для пересдач и хвостов
- **рекомендательная** — система предлагает следующий пакет по результатам предыдущего
- **адаптивная** — ветки выбираются динамически

**Во всех случаях узлы графа — одни и те же пакеты.** Никакого дублирования контента под разные пути. Никаких форков курса под разные категории студентов.

---

## 3. Где это работает в индустрии

Атомарность как фундамент адаптивности — доказанный паттерн, хотя в академической среде применяется редко.

| Платформа | Атомарная единица | Граф |
|---|---|---|
| **Khan Academy** | Навык (skill) | Граф зависимостей навыков, рекомендательная система |
| **Duolingo** | Урок | Дерево курса, адаптивные ветки (Duolingo Max) |
| **Brilliant.org** | Концепт/упражнение | Переиспользование между курсами |
| **Knewton, ALEKS, Smart Sparrow** | Learning object | Полностью адаптивный движок |
| **Anki, RemNote** | Карточка/note | Граф связей, spaced repetition |

Принципиально: **адаптивность невозможна без атомарности**. Если курс монолитом — невозможно подмешивать материал на лету, перестраивать траекторию по результатам, рекомендовать следующее.

Coursera/edX/Moodle-style курсы — это про **доставку готовых программ**, не про адаптивное образование. Они не атомарны, потому что их не для этого делали. Для академической программы по РПД они подходят, для индивидуальной траектории — нет.

## 4. Контекст применения

### В вузе — режим РПД

В рамках академической программы:

- учебный процесс регламентирован: ФГОС → ОП → учебный план → РПД → тематический план
- траектория студента предсказуема и утверждена методически
- атомарность работает «незаметно» — студент идёт по плану, не выбирая

**Атомарность здесь даёт не вариативность для студента**, а:

- переиспользование пакетов между ОП (одна тема — один пакет на все направления)
- независимое обновление пакетов и РПД
- аккуратное версионирование (изменился пакет — не разваливается РПД)
- упрощённое управление академической разницей при переводах
- возможность боковых пакетов для углубления (для сильных студентов / олимпиадников)

### В ДО/ДПО — режим траектории

В дополнительном образовании ограничения существенно слабее:

- нет ФГОС в академическом смысле (есть профстандарты, но они мягче)
- программа собирается под цель учащегося, не под унифицированный план
- продолжительность вариативна (курс выходного дня / полугодовая программа)
- предварительная подготовка участников разная

**Здесь атомарность раскрывается полностью:**

- **программы собираются из пакетов под цель** — «хочу освоить инструмент X для задачи Y», система предлагает набор пакетов
- **индивидуальные траектории** — стартовый ассессмент → подбор пакетов под пробелы
- **дозированное обучение** — можно проходить один пакет в неделю, можно интенсивом
- **переиспользование между программами ДПО** — пакет, разработанный для одной программы, легко включается в другую
- **mixing с вузовскими пакетами** — слушатель ДПО может пройти отдельные пакеты основной академической программы
- **глубокое погружение vs пройти дальше** — система или сам учащийся выбирают, идти в боковой пакет или к следующему по основной линии

### В ДО (детское/доп. образование для школьников)

Аналогично ДПО, с поправкой на возрастные особенности:

- кружки и секции — собираются под интерес ребёнка
- индивидуальные траектории по результатам
- родитель/педагог формирует программу из доступного пула пакетов
- участие в олимпиадах и конкурсах — отдельные пакеты, встраиваются в общую траекторию

## 5. Что атомарность даёт архитектурно

### 1. Траектории как первоклассная сущность

Тематический план РПД — **одна из траекторий**, формализованная регулятором. Другие траектории — это просто другие списки ссылок на те же пакеты. Никаких параллельных копий контента.

### 2. Переиспользование между программами

Пакет «Нормализация 3НФ» нужен на ИВТ, на ПИ, на ИБ, в программе ДПО «Базы данных для аналитиков». В монолите — копия в каждом курсе. В атомах — **ссылка из каждой программы на один пакет**.

### 3. Адаптивный контроль глубины

- студент прошёл базовый пакет, набрал высокий балл → углублённый пакет по той же теме
- не прошёл → пакет повторной подачи с другой методикой
- пограничный результат → дополнительный пакет-практикум

**Невозможно с монолитом**, потому что в монолите глубина зашита автором при создании курса.

### 4. Композиция программ из чужих пакетов

Кафедра X разработала отличный пакет по теме Y. Кафедра Z включает его в свой учебный план **ссылкой**, не копией. Обновился оригинал — обновился у всех. Граничный случай — модель **Wikipedia для образования**: общий пул атомов, разные сборки.

### 5. Поддержка непрерывного образования (LLL)

Студент окончил вуз, через 5 лет хочет освежить тему — заходит, берёт пакет, проходит. **Не нужно искать «модуль 7 курса 2018 года»**. Пакет жив сам по себе, пока РПД, к которой он привязан, актуальна. Это уже не про студента-в-программе, это про **человека-в-знании**.

### 6. Аналитика поведения сильно богаче

«Студент прошёл пакет A, после него выбрал пакет B (а не C, рекомендованный)» — это **сигнал о предпочтении/понимании**. В монолите такого сигнала нет: там нет выбора, есть только прогресс-бар. В графе атомов — поведение студента порождает данные для улучшения рекомендаций и для исследования учебного процесса.

### 7. Естественная стыковка с edu-framework

Атомарность пакетов согласована с философией edu-framework: открытые JSON-форматы, машиночитаемые домены, переиспользуемые единицы. Граф пакетов — естественное продолжение графа доменных сущностей.

## 6. Это переопределяет роль Univerkon

В монолитной парадигме Univerkon — **«система управления учебным процессом по РПД»**: оператор данных деканата, контингента, расписания.

В атомарной парадигме Univerkon становится **графом знаний дисциплин + механизмом навигации по нему**:

- пакеты — узлы
- учебные планы и РПД — формально утверждённые маршруты по графу
- индивидуальные и адаптивные траектории — другие маршруты
- учебные события (D04) — фиксация прохождения узла конкретным студентом в конкретном маршруте

**РПД-режим становится режимом по умолчанию, а не единственным.**

Это **сильно расширяет горизонт Univerkon**:

- вузовская академическая программа
- программы ДПО на базе вуза
- программы ДО при вузе (профильные школы, малые академии)
- адаптивные траектории внутри программ
- индивидуальные образовательные маршруты
- непрерывное образование выпускников

Все эти сценарии — **варианты обхода одного и того же графа пакетов**, не отдельные системы.

## 7. Что меняется в архитектурных вопросах

С учётом этой оптики некоторые открытые вопросы получают новый вес:

### Получают приоритет

- **Граф связей между пакетами как отдельная сущность.** Кто его держит — Univerkon? Отдельный сервис? Часть метаданных пакетов? Какова модель связей (prerequisite, continuation, deepening, alternative, application)?

- **Движок траекторий.** Кто строит маршрут — система автоматически? педагог вручную? студент с поддержкой? В MVP — только статичные траектории из РПД. Адаптивные — позже.

- **Launch context расширяется.** LMS сообщает пакету не только «кто запустил», но и:
  - в составе какой траектории
  - это базовый проход или углубление
  - после какого пакета пришёл студент
  - какие смежные пакеты доступны
  - возможна ли смена траектории

  Пакет может адаптировать введение под контекст: «ты уже изучил предыдущую тему, повторим только ключевое», «это углублённый пакет, базовый уже пройден».

### Упрощаются

- **Версионность пакета vs РПД.** РПД — список ссылок на пакеты с их версиями. Сменилась РПД — изменился список ссылок. Пакеты живут своей жизнью. Bitemporal-логика D06 опирается на этот граф ссылок, а не на содержимое пакетов.

## 8. Что это значит для MVP

Атомарность как фундамент закладывается **с первого дня**, даже если в MVP используется только режим РПД.

**Что делаем в MVP:**

1. Пакеты атомарны и переиспользуемы — это базовый принцип
2. Граф связей — **в виде минимальных метаданных пакета** (prerequisites как ссылки на другие пакеты)
3. Траектории — только из РПД (тематический план как линейный список пакетов)
4. Launch context — базовый (без сложной информации о траектории)

**Что откладываем:**

1. Движок альтернативных траекторий
2. Адаптивные ветки и рекомендательная система
3. Программы ДПО/ДО на базе того же пула пакетов
4. Композиция программ из чужих пакетов разных кафедр

**Принципиальное:** архитектура с самого начала **готова к развитию в сторону графа и траекторий**. Атомарность даёт это бесплатно — нужно лишь не закрыть её преждевременной оптимизацией.

---

## 9. Аргумент наружу

Если формулировать на одну фразу для коллег и руководства:

> **«Один пакет — одно занятие» — это не дробление контента, а фундамент для непрерывного и персонализированного образования. Сегодня мы используем эти пакеты для академических программ по РПД. Завтра — для ДПО, ДО, индивидуальных траекторий, рекомендательного обучения. Без переписывания контента, потому что контент уже атомарен.**

Это **разворачивает разговор о пакете**: с «мелкая удобная единица для деканата» на **«инфраструктурное решение, открывающее новые форматы обучения»**.
